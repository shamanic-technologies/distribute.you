# Project: MCP Factory

DFY (Done-For-You), BYOK (Bring Your Own Keys) automation platform built on the Model Context Protocol. Users provide a URL + budget, the platform handles lead finding, outreach, email generation, and reporting.

The prior note here claiming `cd apps/dashboard && npx next lint` is "clean 0/0" was stale and wrong.)

**RTK summarizes `next build` output — don't trust the summary.** The rtk hook rewrites `next build` and replaces real stdout with a one-line `Errors: N | Warnings: M` (plus a fake sub-second `Time:`). The count can be WRONG — observed a phantom `Errors: 1` when the real run was clean. To verify a real `next build`: bypass rtk with `rtk proxy -- npx next build` (the `--` is required, else rtk's own parser eats flags). Trust the real log + process exit code, never the rtk one-liner. (`next lint` no longer exists — see above — so this only applies to `next build` now.)

**Shared workspace packages must be built before app tests/build.** Vitest + Vite resolve workspace deps via their `dist/` (per `package.json` exports), so an unbuilt `shared/*` package surfaces as `Failed to resolve entry for package "@distribute/<name>"` in unrelated test files. Run `pnpm -r build` (or `pnpm --filter @distribute/<name> build`) once after `pnpm install` or after pulling changes that touch `shared/`.

**Source-substring page tests: a `not.toMatch(/X/)` guard trips on the fix's own explanatory comment.** The dashboard's page tests (`expert-quote-outreach.test.ts`, `feature-quote-pages.test.ts`, `pr-expert-public-report.test.ts`, …) read a page's source as a string and assert substrings. When you add a regression guard like `expect(content).not.toMatch(/score \* 100/)`, a code comment that *explains* the bug ("don't do `score * 100`") contains the very pattern the test forbids → the test fails against your own correct fix. Defense: reword the fix's comment so it doesn't contain the forbidden literal (e.g. "multiplying by 100 rendered 9500"), OR run the assertion against a comment-stripped copy — `pr-expert-public-report.test.ts` already ships a `stripComments(src)` helper for exactly this. Incident 2026-05-31 (distribute.you#1235, DIS-112): the `score * 100` regression guard red-flagged the fixed `ScoreBadge` because the new comment literally wrote `` `score * 100` rendered "9500" ``; reworded the comment, green.

**Run the dashboard test suite locally before pushing a dashboard change — `tsc` alone is not enough.** Many dashboard tests are source-substring guards that assert a page/lib contains a specific expression or copy string (e.g. `expect(content).toContain("budgetCents > parseFloat(account.balance_cents)")`, `toContain("Add a brand")`). When you intentionally change asserted code or copy, `tsc --noEmit` stays green but the guard goes red in CI — and on an auto-merge PR that just silently blocks the merge until you notice. Always run `pnpm --filter @distribute/dashboard test` (fast, ~3s, source-only — no shared build needed for the substring suites) before `git push` on any dashboard PR, and update the matching guard in the same commit. Incident 2026-05-31 (distribute.you#1252): shipped tsc-clean but CI `test-dashboard` failed twice — the budget-vs-balance guard asserted the old `budgetCents > parseFloat(...)` expression (changed to a ceil'd-balance check) and `multi-brand-campaign.test` asserted "Add a brand" (renamed "Add a co-brand"). A local run would have caught both before the first push.

## Release flow (distribute.you specifics)

This monorepo does NOT use `release.sh hotfix` (the user-level release script targets Railway-deployed services with semver tags). Vercel deploys this repo on every `main` merge — no tag, no bump.

- **Hotfix** → branch from `origin/main`, PR target `main`, ship with `gh pr merge --auto --squash` (or hold for deploy-ordering blockers).
- **Bugfix / Feature** → branch from `origin/staging`, PR target `staging`, ship with `gh pr merge --auto --squash`. Promotion to `main` happens via the existing staging→main PR flow.
- **Observed exception (`apps/dashboard/` and `apps/landing/` only):** dashboard-only and landing-only PRs have been merging straight to `main` in recent practice (see #1146–1151, #1175, #1186). When touching ONLY those two app dirs, follow the convention of the last 5 merges on the file you're editing; when in doubt, ship to `staging` per the rule above. Cross-cutting changes (shared/, packages/, multi-app) MUST go through staging.
- **Cut a NEW branch off `origin/main` before each change — never commit onto the just-merged branch.** After a `gh pr merge`, the workspace stays checked out on the merged branch; in a rapid iterate-and-ship loop it's easy to start editing + `git add`/`commit`/`push` on it. That branch's PR is already merged, so (a) it violates "never push to a merged branch" and (b) re-branching off it later double-includes its squashed content in the next PR diff. ALWAYS run `git fetch origin main && git checkout -b KevinLourd/<topic> origin/main` as the first step of a new change. Recovery if you slip: `git checkout -b <new> origin/main && git cherry-pick <your-sha>` (main already has the merged content, so the cherry-pick applies clean as a single-commit diff). Note: a branch created via `checkout -b X origin/main` tracks `origin/main`, so `git push -u origin X` can trip the protect-main hook — push with an explicit refspec `git push origin HEAD:refs/heads/X`. Incident 2026-05-31 (distribute.you#1248): committed the workflow-test-emails feature onto the already-merged `sales-workflow-picker` branch; recovered via cherry-pick onto a fresh branch.
- **Base-branch prerequisite check (MANDATORY before `git checkout -b … origin/staging`).** If the feature builds on code that recently merged to `main` but has not yet been promoted to `staging` (typical for dashboard work given the exception above), the new branch MUST be cut from `origin/main` instead. Cheap check: `git log origin/main ^origin/staging -- <prerequisite-path>` — if it returns any commits, your prereq is missing on staging. Skipping this check silently produces "files don't exist" failures mid-execution and forces a `git stash + branch-recreate` recovery. Incident 2026-05-25 (distribute.you#1186): cut journalist manual-reply branch from staging by default; leads-side `EditLeadStatusModal` + `useManualQualifications` files were on main (#1175) but absent on staging. Required mid-execution recut to `origin/main` after `Glob` for the prerequisite returned no matches.

### Missing Backend Fields

If the dashboard needs a field, endpoint, or capability that the backend doesn't provide, NEVER work around it client-side (regex, slugifying, name-derivation, aggregation heuristics, etc.). ### Verify backend shape before writing client types

Before declaring a typed dashboard helper for any backend endpoint (`listX`, `getX`, response interface, Zod schema), fetch the actual returned shape via the `api-registry` / `api-registry-staging` MCP. Never invent a type from "what feels right" or copy from an aspirational PR description — the deployed openapi is ground truth. If you want a richer shape, file a backend request and block (see "Missing Backend Fields"). Skipping this produces page crashes like `Cannot read property 'X' of undefined` because `data.X` does not exist on the real response.

Incident 2026-05-29 (distribute.you#1215): the PR Expert quote-opportunities GENERATE button produced low-quality pitches (no crash — the call *succeeded*, the output was just garbage). Root cause is a NEW class of wire-shape rot that TypeScript/Zod CANNOT catch: **template/prompt-style endpoints take a free-form `variables: Record<string, unknown>` passthrough body**, so any JSON validates at the wire boundary — but the variable-NAME contract is published separately via the template's `.variables` metadata (`GET /v1/content/platform-prompts?type=<template>`). The deployed `expert-quote-pitch` template (DIS-52) declares exactly `{{brand}}`, `{{request}}`, `{{additionalContext}}`; the dashboard sent flat keys (`spokesperson`/`expertiseTopics`/`opportunityText`/`mediaOutlet`/…) with ZERO name overlap, so every `{{...}}` rendered empty and the model got no grounding. A second consumer (public-report `/draft` route) rotted identically — its `OPPORTUNITY_VARS` whitelist matched no current template var, so it brand-extracted `{{request}}` (the journalist's question) from brand-service. **Lessons**: (a) for any endpoint whose body is a `variables`/`inputs` passthrough rendered into a stored template, the source of truth is the template's published `.variables` list, NOT the old code, NOT a response schema, NOT "what feels right" — fetch it live and map your payload keys 1:1 before trusting generation output. (b) Free-form passthrough = no compile-time / safeParse safety net; the failure mode is *silent bad output*, not a crash, so it survives every type check and every "page loads" smoke test. When a generation feature ships "works but bad quality", suspect a variable-name contract mismatch FIRST. (c) A template refactor in the upstream service (DIS-52 dropped scalar schemas for `Record<string,unknown>` flex) silently invalidates EVERY caller's variable-name assumptions at once — audit all callers (`rg` for the generate path + the template type) when an upstream template's `.variables` changes, same as for an endpoint refactor. Fix centralized the assembly in `apps/dashboard/src/lib/quote-pitch-variables.ts` so both consumers share one mapping.

Incident 2026-05-29 (distribute.you#1221): the per-feature prompt editor's Save (`savePromptAssignment`) threw `[dashboard] savePromptAssignment: invalid response shape` on every *successful* fork — the operator saw a red error on a save that actually persisted (reopen via GET showed the new fork). Root cause is a distinct wire-shape sub-class: **one Zod schema reused for both the GET (read) and PUT (write) response of the "same" resource.** `#1216` declared `PromptAssignmentSchema` (with `isDefault` required) and parsed BOTH `GET /v1/content/prompt-assignments` (returns `isDefault`) and `PUT` (does NOT — confirmed prod + staging registry; the write DTO is deliberately narrower because post-fork `isDefault` is definitionally false) with it. The shared schema's required `isDefault` turned every PUT 200 into a safeParse failure. **Lessons**: (a) a write endpoint's response DTO is frequently NARROWER than its read sibling (omits computed/derived fields like `isDefault`, `updatedAt`, server-resolved counts) — give each verb its OWN response schema, each verified independently against the registry (`get_endpoint_details` lists `required` per response). Reusing one schema across read + write couples them, and a single field present on read but absent on write converts every successful write into a thrown "invalid response shape". (b) The DIS-74 safeParse rule is correct and was followed — but note its dual edge: too-LOOSE a schema lets bad data render and crash (the #1213 failure mode); too-STRICT a shared schema converts SUCCESS into a false error (this incident). safeParse is only as good as a per-endpoint shape; verify read AND write shapes separately. (c) When a write succeeds server-side but the client surfaces an error, suspect a response-schema mismatch on the write path before suspecting the write itself. Fix: `SavePromptAssignmentResponseSchema = PromptAssignmentSchema.omit({ isDefault: true })`, return `{ ...parsed, isDefault: false }` (documented post-fork invariant, not a fallback).

Incident 2026-06-01 (distribute.you#1268): the sales workflow-picker "Run test" button appeared dead — click created the test campaign (verified in campaign-service DB) but the modal never showed a spinner, "Generating emails…", or any email row. Root cause is a wire-shape sub-class that NEITHER TypeScript NOR Zod catches: **a free-`string` status field whose openapi `description` carries a stale "e.g." EXAMPLE value that a dev copied as a real literal.** The api-service `GET /v1/campaigns/{id}` schema declares `status: { type: "string", description: "Campaign status (e.g. 'active', 'stopped')" }` — a free string, NO enum. The dashboard gated its test poll + spinner on `status === "active"`, but campaign-service only ever emits `"ongoing"` (running) / `"stopped"` (terminal) — **569 prod rows, zero `"active"`**. The `"active"` literal came straight from the doc's parenthetical example. Result: `runningTestIds` was always empty → React Query `enabled:false` → emails never fetched → silent dead button (no crash, no error — survives every type-check and "page loads" smoke test, same failure signature as the #1215 variable-name class). **Lessons**: (a) an openapi `description` with `(e.g. 'x', 'y')` on a free-`string` field is NOT an enum and its example value can be stale/wrong — never copy it as a literal. Verify the ACTUAL value set against DB ground truth (`SELECT status, count(*) FROM campaigns GROUP BY status`) or live data, per "live > source". (b) For status/state comparisons, prefer gating on the TERMINAL value (`status !== "stopped"`) over an assumed running label — terminal states are fewer, more stable, and a new transient running label (`queued`, `pending`) won't silently fall out of the "running" set. (c) Same defense as #1215: when a feature "fires but the UI does nothing", suspect a string-literal contract mismatch (status enum, variable name) before suspecting the call itself.

### No Fallbacks — Fail Visibly

NEVER add fallback logic (|| alternatives, silent defaults, graceful degradation) when data is missing or doesn't match. Instead, log a clear `console.error` with the mismatched value and context so the bug surfaces immediately. If a required field is absent, show an error UI — don't hide the problem. This applies everywhere: lookups, field resolution, display logic.

**Exception — Vercel build-time prerender steps.** During `next build` static generation (sitemap.ts, generateStaticParams, dynamic OG routes), an unhandled throw aborts the entire deploy, not just the failing page. When the failure mode is data-shape (missing table, unreachable DB on a not-yet-migrated preview branch) rather than logic, prefer `try / console.error(loud) / continue with empty result` over throwing. The fix path is still to repair the data layer; the catch+log keeps the rest of the site shippable while you do. This applies ONLY to build-time prerender — runtime requests still follow the fail-loud rule. Incident 2026-05-22 (distribute.you#1120): sitemap.ts threw `relation "blog_articles" does not exist` during `/sitemap.xml` prerender, blocked the entire landing deploy across 16 pages. The existing `process.env.DATABASE_URL` guard checked the URL, not the schema; one missing table killed everything.

### React Query mutations: write the response to the cache, don't just invalidate

When a mutation returns the fresh entity (e.g. POST `/campaigns/{id}/stop` returns `{ campaign }` with the new status), write it into the single-entity cache via `queryClient.setQueryData(["entity", id], data)` instead of (or in addition to) `invalidateQueries`. The downstream GET endpoint can return 5xx, and with `placeholderData: keepPreviousData` an invalidate-then-failed-refetch leaves the cache holding the stale pre-mutation row. The user sees the button revert / status unchanged and concludes the click did nothing.

`invalidateQueries` is still correct for list caches the mutation cannot rebuild on its own (`["campaigns"]`, `["leads"]`, etc.) — there the upside of a fresh list outweighs the risk of a flaky GET, and the mutation has no full list to write.

Incident 2026-05-20 (distribute.you#1090): `useStopCampaign` only invalidated `["campaign", id]`. With api-service `GET /v1/campaigns/{id}` returning 500, single-click Stop appeared to do nothing — user had to click twice. Fix: `setQueryData(["campaign", id], data)` on success.

### Dashboard data fetching — ONE framework (homogeneous SWR via React Query v5)

Stack: TanStack React Query v5 on every client component. No SWR. No mixing. Server components for the shell only; everything interactive uses `useAuthQuery` / `useMutation`.

**Global config (`apps/dashboard/src/lib/query-provider.tsx`) — single source of truth:**
- `staleTime: 60_000` (1 min). Default `0` causes flash on every mount/window-focus.
- `gcTime: 5 * 60_000`.
- `placeholderData: keepPreviousData` (GLOBAL). Stale data stays on screen during refetch — true stale-while-revalidate. Never re-set per query.
- `refetchOnWindowFocus: true`, `refetchOnReconnect: true`.
- `retry: 1` (queries), `retry: 0` (mutations).

**Shared poll cadences (`apps/dashboard/src/lib/query-options.ts`):**
- `pollOptions` → 5s (default for active pages).
- `pollOptionsSlow` → 10s (quote-pitches, quote-requests, press-kits).
- `pollOptionsSlower` → 30s (visibility-runs, low-frequency).
- Import + pass to `useAuthQuery(..., pollOptions)`. Never declare a local `const pollOptions`.

**V5 status flags — render rule:**
- `isPending` → no data ever → render SKELETON.
- `isLoading` (= `isPending && isFetching`) → equivalent for initial load → render SKELETON.
- `isFetching && !isPending` → background refetch with cached data → render CONTENT, never skeleton.
- Skeleton condition is exclusively the "no data" case. Background refetches are SILENT — no top-bar, no spinner, no flash. Modern SWR philosophy: stale-while-revalidate is invisible by design (matches Linear, Vercel, GitHub). User-initiated refresh / mutations still surface via button-scoped spinners.

**Page composition:**
- Shell + nav + header render INSTANTLY (no query gates them).
- Each card/section owns its own `useAuthQuery` + skeleton. Parallel, never cascaded unless data truly depends.
- Skeleton must match the real layout (heights, columns) — zero layout shift on resolve.
- Spinner only for button-scoped actions (Save, Send). Never for content.
- `<Suspense>` is NOT used in dashboard pages (use `isPending` checks). Reserved for public marketing pages only.

**Mutations (extends incident #1090):**
- If the mutation returns the new entity → `queryClient.setQueryData(["entity", id], response)` FIRST.
- THEN `invalidateQueries({ queryKey: listKey })` for any list cache the mutation cannot rebuild.
- Return the invalidation promise from `onSuccess` so the button stays loading until lists refresh.

**Forbidden patterns:**
- Local `const pollOptions = {...}` (use shared module).
- `refetchIntervalInBackground: false` (it's the v5 default — drop).
- `placeholderData: keepPreviousData` in per-query options (now global default — drop).
- Skeleton on `isFetching` when data exists.
- `<Suspense>` for any dashboard content.
- **A unified gate that ANDs a WARM section together with a COLD sibling (the "everything waits on the slowest endpoint" gate).** A warm-cache section (feature names in `["features"]`, anything with `staleTime` minutes) MUST NEVER be held behind a cold sibling query — that is the failure #1257 fixed. This is NOT a blanket ban on revealing several sections together: see the "Coordinated reveal" rule below — within ONE coherent group whose queries are all cold, revealing them in unison (barrier + latch) is the prescribed pattern. The rule is about GROUP GRANULARITY, not about forbidding unison: group at the finest coherent level, barrier WITHIN a group, independent ACROSS groups, and never fold a warm section into a cold group's gate. Per-card sub-data (e.g. campaign stats inside a feature card) is its own finest-grain group — own skeleton, own query. Incident 2026-06-01 (distribute.you#1257): brand-overview `allReady` ANDed 8 queries (brand, campaigns, batchStats, 5× outcome counts); feature names paint instantly from the warm `["features"]` cache in the sidebar, but the body held them hostage behind the slowest outcome endpoint → cards showed skeletons for seconds. Fix: render the features grid from the warm cache immediately, scope the outcomes-count gate to the 5 outcome queries only (their own coordinated group), give per-card stats their own skeleton. (DIS-96 introduced the mega-gate under a "no staggered wave" banner; #1257 split it by group; the granularity principle is the unifier.)

**Coordinated reveal (synchronized groups + monotonic latch).** Reveal timing is a SEPARATE concern from query ownership (each section still owns + fires its own query in parallel, per "Page composition"). When several sections form one coherent visual group whose data is all cold, they should reveal **together in one paint** (no card-by-card cascade) and then **stay revealed** — a background poll / Clerk token rotation / transient query error must never send a shown group back to a skeleton. The single source of truth is `apps/dashboard/src/lib/use-coordinated-reveal.ts`:

- `useCoordinatedReveal(readyFlags: boolean[]): boolean` — **barrier** (returns false until every flag is true at least once) + **monotonic latch** (once true, stays true for the mount, regardless of later flag flips). The latch is the reveal-layer analog of React Query's `keepPreviousData`: `keepPreviousData` keeps each query's *data* on screen during refetch; the latch keeps the *group's reveal decision* from reverting. Both are needed — a gate written on live `!isLoading` / `data !== undefined` is sticky in steady state but still flips on a cache-clear / first-load rotation; the latch makes "shown once ⇒ shown forever (this mount)" unconditional.
- One readiness flag per query in the group: `data !== undefined`, or `!isPending`. **A *disabled* React Query stays `isPending: true` forever** — gate its flag behind its own `enabled` condition (`!enabled || !isPending`), or it pins the whole group in skeleton state.
- **Group at the finest COHERENT level, and NEST groups coarse→fine.** A surface has a hierarchy of reveal groups, each revealed in one paint, each nested inside the coarser one. On a feature page: (1) the **body group** (campaigns + stats + cost) reveals together; (2) the sidebar **nav-items group** (every link) reveals together; (3) the sidebar **badge-numbers group** (Outcome counts) is a finer sub-group inside the nav group. Coarser group resolves → its skeleton structure appears → finer sub-group fills in. Never widen a group to span a warm + cold pair (see the forbidden-patterns rule above).
- **Gate the barrier on its INPUTS being loaded FIRST (`defsReady`), before any per-query flag.** When a group's queries are *conditionally enabled* (`enabled: someList.includes(x)` where `someList` comes from another query — feature defs, an entity registry, a parent id), there is a first-paint window where the enabling input hasn't loaded, so `someList` is empty, EVERY query is disabled, and every `!enabled || !isPending` flag is `true` → the barrier passes **instantly and latches an EMPTY group** (no skeleton, real values then trickle in one-by-one — the exact bug it was meant to prevent). Always prepend an inputs-loaded flag: `useCoordinatedReveal([defsReady, ...perQueryFlags])` where `defsReady = !featuresLoading && !registryLoading` (or whatever resolves the `enabled` conditions). Incident 2026-06-01 (#1260): `FeatureLevelSidebar` badges revealed empty because `entityNames` was `[]` until the feature loaded.
- **Gate each section on the query that produces ITS rendered data — never on a context's coarse `loading`.** A single-section page still staggers if its skeleton watches the wrong flag. A shared context (e.g. `useCampaign`) often exposes ONE `loading` that tracks only its primary query (the campaign), while the list a page actually renders (`emails`, `leads`) is a SEPARATE query that resolves later. Gating the page's skeleton on `loading` hides it the instant the campaign resolves → the still-fetching list flashes its empty state, then rows pop in. Fix: the context must expose per-entity loading (`emailsLoading`, `leadsLoading`) and each page/badge gates on the flag for the data it shows. This bug is invisible to a "how many sections does the page have" audit — it's a wrong-flag bug, not a multi-section bug. Incident 2026-06-01 (#1264): `campaigns/.../{emails,companies,leads}` + the campaign sidebar badge group all gated on `useCampaign().loading` (campaign-only) instead of `emailsLoading`/`leadsLoading`.
- **Sidebar primitives.** Reveal a whole nav level together by gating its body on `defsReady` and rendering `SidebarNavRowSkeleton` rows until then (else static top items paint first and data-dependent items pop in later — #1261). Reveal a group of count badges together by passing `badgePending={!badgesRevealed}` to `SidebarLink` (renders a skeleton pill until the badge group settles). Both live in `context-sidebar.tsx`.
- Call it unconditionally at component top level (it uses `useRef` — hooks rule); the ref resets naturally on unmount, so a freshly-mounted page barriers again from its own cache state.

First landed 2026-06-01 (distribute.you, `sales-cold-email-outreach` feature page + `FeatureLevelSidebar`, PRs #1259/#1260/#1261; DIS-133): page body, sidebar nav items, and sidebar badge numbers each reveal as one latched group. **This is now the standard for every multi-query dashboard surface** — page bodies AND all data-dependent sidebar levels (`BrandLevelSidebar`, `CampaignSidebar`/`McpSidebar`, `WorkflowLevelSidebar`, `AppFeatureLevelSidebar`). When adding/editing such a surface, identify its reveal groups, gate each with `useCoordinatedReveal` (inputs-loaded flag first), and render matching skeletons. Static-only nav levels (no data-dependent items) need no gate.

Root cause was a combination of (a) no global `placeholderData: keepPreviousData`, so v5 dropped cached data to `undefined` on every refetch, and (b) 21 page files declaring redundant local `pollOptions` constants that drifted (3 included `keepPreviousData`, 18 omitted it). Fixed by lifting `keepPreviousData` to the global default + collapsing all pollOptions to a 3-variant shared module + dropping all `refetchIntervalInBackground: false` (default in v5).

### Monotonic status latch — status-tab tables must not flap on a poll (`useMonotonicStatuses`)

The third member of the "don't revert resolved state on a transient" family, after React Query's `placeholderData: keepPreviousData` (keeps a query's *data* across refetch) and `useCoordinatedReveal` (keeps a group's *reveal decision* across refetch). This one keeps a **per-row derived bucket** across refetch — and it is the ONLY one of the three that addresses a *valid CHANGING success* (not a skeleton-revert, not an error).

**The failure class it fixes.** A page that buckets entities into mutually-exclusive **status tabs** and renders ONE tab at a time derives each entity's tab from a delivery/engagement overlay that is re-fetched on every poll (leads: `getLeadConsolidatedStatus` reads the email-gateway booleans `replied/clicked/opened/delivered/sent/contacted`; journalists/outlets: `deriveDisplayStatusFrom{Booleans,Counts}` off `@/lib/outlet-status`). When that overlay transiently drops on a poll (status arrives under a different shape — e.g. lead-service `flattenBrandStatus` ignoring the `byCampaign` breakdown — or empty), every entity falls from "Delivered/Opened/Replied" back to the raw "Processing", **leaves the tab the user is viewing**, and the table appears to empty, then repopulates next poll. `keepPreviousData` and coordinated-reveal CANNOT fix this: the response is a valid 200 whose content legitimately changed, so the cache correctly updates — the bug is that a transient downgrade is indistinguishable from a real one at the wire. (Confirmed it is NOT a query/proxy/cache-wipe bug: the page query + global `keepPreviousData` retain data, the `/api/v1` proxy fails loud on non-2xx, the shell layout is latched (#1255), the cache is never wiped on a poll, and lead-service returns either full rows (200) or 500 — never a successful-empty.)

**The primitive.** `apps/dashboard/src/lib/use-monotonic-status.ts` → `useMonotonicStatuses(entries: {id, status}[], priority: readonly string[], label?)`: a per-mount latch keyed on the page's OWN "most-advanced-first" priority array (`LEAD_STATUS_ORDER`, or `STATUS_PRIORITY` from `outlet-status.ts`). Outreach engagement is **append-only** (`contacted → sent → delivered → opened → clicked → replied` never reverses), so a less-advanced status on a later poll is by domain definition a stale/partial read. Keep the most-advanced status seen this mount; `console.error` the suppressed downgrade so the upstream inconsistency stays visible (**fail-loud, NOT a silent fallback** — it encodes a real invariant and surfaces nothing false). Pure reducer `nextMonotonicStatus` is exported + unit-tested (mirrors `nextRevealState`). The returned Map is the **single source** the tabs, the row badge, and the detail panel all bucket on — so they never disagree.

**The rule.** Every **client-polled** status-tab surface MUST latch bucket membership via `useMonotonicStatuses`. A static (ISR / server-component) status table is NOT susceptible — it renders once per cache window, no poll re-shuffles it — so it needs no latch (the public report leads tree is excluded for exactly this reason; hooks can't run there anyway). When adding a new status-tab page, build `{id, status}` from the rendered list, latch with the page's priority array, and read the latched status everywhere you'd otherwise call the raw status fn.

First landed 2026-06-02 (distribute.you#1270, DIS-149): the sales-cold-email-outreach leads table flashed empty every ~5s. Wired into all 9 client-polled status-tab surfaces (feature/campaign/brand × leads/journalists/outlets) + added the missing `safeParse` on `listBrandLeads`/`listCampaignLeads`. Backend root (lead-service `flattenBrandStatus` + email-gateway overlay consistency) tracked as the DIS-149 cross-repo follow-up; the latch makes the UI robust regardless of which poll the overlay drops on.

### Persisted query cache — leave a page and return shows content INSTANTLY, never a skeleton (`PersistQueryClientProvider`)

The **4th and outermost** member of the "don't revert resolved state on a transient" family — and the only one that survives leaving the page entirely. The first three all operate on a **warm in-memory cache** and protect against *refetch* flicker:

1. `placeholderData: keepPreviousData` — keeps a query's **data** across refetch.
2. `useCoordinatedReveal` — keeps a group's **reveal decision** across refetch.
3. `useMonotonicStatuses` — keeps a row's **bucket** across refetch.

**None survive cache EVICTION.** `gcTime` (was 5 min) drops a query's data once it's been inactive that long, and a full reload / new tab starts a brand-new `QueryClient` with an empty cache. So after navigating away for >gcTime, or reloading, the in-memory cache is **gone** — `data === undefined`, the coordinated-reveal barrier has nothing to reveal, and the page cold-loads a full-screen skeleton for the duration of the refetch. That is the "I left the page and came back, everything disappeared for a few seconds" bug. The first three layers cannot fix it because there is no data left in memory to keep.

**The failure class it fixes.** Leaving any dashboard page and returning (long internal nav crossing gcTime, OR browser reload / new tab) repainted every section as a skeleton, then refilled after the refetch landed. The cache was correct; it had simply been evicted, and nothing restored it.

**The fix (industry consensus — TanStack + SWR/Vercel both ship cache persistence as THE answer; Linear goes further with a local-first sync engine).** Persist the React Query cache to client storage and restore it on mount, so any return serves the last-known content instantly while revalidating silently in the background. Single global change in `apps/dashboard/src/lib/query-provider.tsx`: wrap the app in `PersistQueryClientProvider` + `createSyncStoragePersister` (localStorage, SWR-style synchronous restore = zero-frame). Because it's the global provider, it fixes **every** dashboard page at once — there is no per-page work for this symptom. Pure helpers live in `apps/dashboard/src/lib/persist-cache.ts` (`shouldPersistQuery` / `persisterStorageKey` / `cacheBuildId`), unit-tested like `nextRevealState`.

**The five load-bearing rules (each one bites if skipped):**
- **`gcTime` MUST be ≥ the persister `maxAge`** (both 24h via `PERSIST_MAX_AGE_MS`). If `gcTime < maxAge`, in-memory GC evicts a query *before* its persisted copy can restore, silently defeating persistence (TanStack hard rule, discussion #5169).
- **Org-scope the storage KEY** (`distribute-dashboard-cache:{orgId}`, `anon` before Clerk resolves). React Query keys are not yet org-scoped (DIS-143), so a single shared bucket would restore org A's data under org B after a reload. Per-org buckets close that vector. `useOrganization()` works inside `QueryProvider` because ClerkProvider is in `(authed)/layout.tsx`, an ancestor.
- **`buster` = build id** (`NEXT_PUBLIC_VERCEL_GIT_COMMIT_SHA ?? "dev"`). A new deploy busts the persisted cache, so a changed data shape never restores stale into new components.
- **`retry: removeOldestQuery`.** localStorage caps at ~5MB; a multi-MB cached list (e.g. our 12MB leads payload incident) would throw `QuotaExceededError`. `removeOldestQuery` evicts the oldest entry and retries until it fits — the page you're on is most-recently-used, so it persists last and survives. This is what makes localStorage (vs IndexedDB) safe at our scale: it degrades, never crashes. (IndexedDB via `createAsyncStoragePersister` is TanStack's pick for >5MB caches, but its async restore reintroduces a micro-flash window — localStorage's synchronous restore is strictly better for the no-skeleton goal, and `removeOldestQuery` neutralizes the cap.)
- **`shouldDehydrateQuery` = success + non-sensitive only.** Replaces TanStack's default (so the `status === "success"` check is explicit) AND excludes secret query roots (`apiKeys`/`byokKeys`/`keySources`) — key material must never touch disk.

**The rule.** Cache persistence is GLOBAL and already in place — do NOT add a second `QueryClientProvider` or a per-page persister. When adding a query whose data is secret, add its key root to `SENSITIVE_QUERY_ROOTS`. When changing `gcTime`, keep it ≥ `maxAge`. The SSR guard (`typeof window !== "undefined"` → undefined storage = no-op persister) keeps it safe in the server render.

First landed 2026-06-02 (distribute.you, `query-provider.tsx` + `persist-cache.ts`): the `ai-visibility-scoring` feature page (and every dashboard page) blanked all sections for a few seconds on return because `gcTime: 5min` evicted the cache. Bumped `gcTime` to 24h + persisted the cache to org-scoped localStorage. The reported page already had `useCoordinatedReveal`; the bug was one layer below it — nothing left in memory to reveal.

### Public marketing pages — SEO + AI-scraper rendering

For any public landing/marketing/docs page (anything under `apps/landing/`, `apps/sales-cold-emails-landing/`, `apps/docs/`), the default rendering strategy is **ISR + `unstable_cache` + edge cache**, NEVER Suspense streaming on indexable content.

Why: Next.js streaming SSR injects content into `<template>` blocks resolved by client-side JS. Googlebot (Chromium-headless) executes JS and sees the resolved content, but **AI scrapers (GPTBot, ClaudeBot, PerplexityBot, Bingbot AI, curl-like crawlers) parse raw HTML only** — they receive the skeleton fallback, not the data. Putting leaderboards, benchmarks, blog lists, stats behind `<Suspense>` kills AI Search indexing for the very content that earns AI Search traffic.

Concrete rules:

1. **No `<Suspense>` on indexable content.** Hero copy, h1/h2, body text, JSON-LD blocks, leaderboards, benchmark tables, blog cards, stats — all SSR-sync in the byte stream. Suspense reserved for user-specific, non-indexable widgets (logged-in dashboards, real-time counters, chat).
2. **Never combine `force-dynamic` with `revalidate`.** `force-dynamic` wins and ISR is silently disabled. Prefer ISR (`export const revalidate = N`) alone; let `unstable_cache` + graceful build-time empty-result handle the missing-config case.
3. **Calling `headers()` or `cookies()` in a page forces dynamic rendering** even with `revalidate` set. Drop them if the page can be ISR-prerendered (resolve hostname-dependent URLs from `NEXT_PUBLIC_*` env vars instead).
4. **Wrap every fetch / DB call used by a public page in `unstable_cache`** with explicit tags. Edge cache (`s-maxage / stale-while-revalidate`) provides instant HTML; `unstable_cache` provides instant data on the rebuild after invalidation. Both stack.
5. **Invalidate via `revalidateTag(tag, profile)`** in the publish/update flow (webhooks, admin forms). Next 16.2+ requires the 2nd arg `profile: 'default' | 'seconds' | 'minutes' | 'hours' | 'days' | 'weeks' | 'max' | CacheLifeConfig`. Pass `"default"` for ordinary invalidations.
6. **`unstable_cache` at module top level breaks vitest** because the wrapping runs at import time before `vi.doMock` can intercept. Mock `next/cache` globally via `apps/landing/tests/setup.ts` (`unstable_cache` → pass-through identity, `revalidatePath` / `revalidateTag` → `vi.fn()`); declare `setupFiles: ["tests/setup.ts"]` in `vitest.config.ts`.
7. **Validate scraper output after any caching change** with `apps/landing/scripts/seo-snapshot.sh [BASE_URL]` (counts `<h1>`, `<h2>`, `<article>`, JSON-LD blocks under a `GPTBot` user-agent) and `apps/landing/scripts/measure-ttfb.sh [BASE_URL]` (warm/cold TTFB per route). Both pre- and post-change snapshots in the PR description.

Incident 2026-05-24 (distribute.you#1153): pre-refactor, `/blog` TTFB was 1.5–2.3s because `force-dynamic` overrode `revalidate = 60` and every visit hit Neon directly. `/benchmarks` was dynamic because `headers()` was called for hostname resolution. `/investors` took 6–8s end-to-end because metrics were `cache: "no-store"`. Three loading.tsx files existed but were never reachable (page components weren't async-suspended). Single PR flipped 8+ routes from `ƒ Dynamic` to `○ Static` ISR with `unstable_cache` + tag invalidation; no Suspense or skeleton was added on any indexable content.

### Feature maturity gating (alpha / beta / ga) — dashboard

Immature features stay in prod (so the founder generates real data with them) but are hidden from non-staff via **PostHog feature flags**, NOT `NODE_ENV` (env-gating couples visibility to deploy — can't flip per-user, can't onboard a beta tester without a redeploy).

Three levels:
- `alpha` → staff only. PostHog flag targeted at the staff person-property `email = kevin.lourd@gmail.com`.
- `beta` → opt-in cohort. PostHog flag targeted at a beta cohort.
- `ga` → everyone. **No flag** — always rendered, never in the registry.

**Single source of truth:** `apps/dashboard/src/lib/feature-gates.ts` — `FEATURE_GATES` maps a surface key → `{ flag, maturity }`. Flag naming convention is `<maturity>-<surface>` (e.g. `alpha-services-crm`).

**Two primitives, reused for every gated surface:**
- `useFeatureFlag(flag): boolean` (`src/lib/use-feature-flag.ts`) — **default-hidden**: returns `false` until PostHog loads flags, so a gated surface NEVER flashes for a non-staff viewer during the async flag fetch. Subscribes to `posthog.onFeatureFlags` so it re-resolves after `posthog.identify(...)` (which is what sets the `email` person-property the alpha flags target — done in `PostHogAuthTracker`).
- `<MaturityBadge level>` (`src/components/maturity-badge.tsx`) — amber `alpha` / violet `beta` pill. Rendered next to a gated surface; since the surface is hidden from everyone else, in practice only staff/beta viewers ever see the badge.

**Pattern:** call `useFeatureFlag` at the top of the component (hooks rule — never inside JSX conditionals), gate the render, attach the badge:
```tsx
const ok = useFeatureFlag(FEATURE_GATES["services-crm"].flag);
// ...
{ok && <SidebarLink item={{ ..., maturity: FEATURE_GATES["services-crm"].maturity }} />}
```

**Graduation needs no redeploy for the audience change** — widen the flag's targeting in the PostHog UI (email → cohort → 100%). Code changes only to relabel maturity, or to fully GA-ify (delete the flag + drop the gate so the surface renders unconditionally).

**Flags are created/edited via the PostHog MCP** (`mcp__posthog__exec` → `create-feature-flag` / `update-feature-flag`), not by hand in code. Adding a new gated surface = add a registry line + create the matching flag.

"Disappear for everyone" (not a per-viewer toggle) is a **hard removal of the JSX**, NOT a flag defaulted-off — a permanently-off flag is dead infra.

### Auth / first-run gating — key on the real product signal, NOT org existence

**Routing/auth decisions belong at the EDGE (middleware/`proxy.ts`) via JWT claims — decided pre-paint, zero data fetch. SWR (React Query) is for DATA freshness, never for routing.** A gate that decides "send the user to onboarding" must (1) key on a signal the platform controls per-user, not on Clerk org existence, and (2) be evaluated server-side before the page paints — a client `useEffect` + `router.push` after a fetch is the flash anti-pattern (dashboard paints, then redirects) and couples the decision to that fetch's latency.

**Don't gate on org existence.** A Clerk org is auto-created at signup (Clerk-native setting or backend `user.created` webhook), so by the time the dashboard loads a brand-new user ALREADY has an active org. `if (!hasOrg) router.push("/onboarding")` is dead on arrival — never fires, onboarding never renders. Incident 2026-05-30 (#1229, DIS-91): produced **0** `/onboarding` pageviews in 60 days.

**Current architecture (DIS-111, #1236).** The durable first-run signal is `org.publicMetadata.onboardingComplete`, set when the first brand is created via an **in-repo server route** (`/api/onboarding/complete` — the dashboard has `CLERK_SECRET_KEY`, so `clerkClient.organizations.updateOrganizationMetadata` works; `publicMetadata` is writable from the dashboard's **server**, just not from the browser). It's surfaced as a Clerk session-token claim (`orgMeta = {{org.public_metadata}}`, configured in Clerk Dashboard → Customize session token) and read at the edge: `proxy.ts` redirects to `/onboarding` when `sessionClaims.orgMeta?.onboardingComplete !== true`. The same `orgMeta`/`email`/`firstName`/`lastName` claims also let the api proxy drop its per-request `currentUser()` round-trip (read identity from the JWT instead).

**Migration checklist for any edge-claim gate (the 3 traps that bite):** (a) **Backfill** every existing entity that should pass the gate BEFORE shipping it — else "claim absent ≠ true" bounces all existing users (here: flag every org that already has ≥1 brand, `apps/admin/scripts/backfill-onboarding-flag.mjs`). (b) **Stale token**: a claim is frozen at JWT mint, so after writing the metadata force a fresh token (`session.getToken({ skipCache: true })`) before navigating, or the next nav loops back on the old claim. (c) **Exempt the loop + in-flight routes**: the gate must skip the onboarding route itself, all `/api/*` (the flow fetches them), and the transient brand-creation hop (`?autoCreate`). Same trap applies to any auto-provisioned entity (org, workspace, default brand) — gate on "has the user COMPLETED the thing?", not "does the container exist?". The superseded client brand-count gate (#1229) is gone; do not reintroduce a client-side routing redirect.

**Readiness gates MUST be monotonic — never blank a mounted subtree on a transient auth-loading flip.** This is the auth-layer analog of React Query's `keepPreviousData`: once a gate has resolved truthy at least once, it must NOT revert to the loading/blank branch when an underlying readiness flag transiently flips back. **Clerk's `isLoaded` (from `useOrganization`/`useAuth`/`useUser`) flips back to `false`, and `organization` can blink `null`, during background session-JWT rotation / revalidation** (~once a minute, plus on focus/reconnect). A render gate written as `show = !isLoading && hasOrg` therefore disappears/reappears the entire gated body on every rotation. Fix = latch: `const resolved = useRef(false); if (!isLoading && hasOrg) resolved.current = true; const show = resolved.current || (!isLoading && hasOrg);` — blank only before the FIRST resolve; the ref resets naturally on sign-out (the layout unmounts). The data-fetching framework (global `placeholderData: keepPreviousData`, page skeletons gated on `isPending && !data`) already handles this for query data — but a SEPARATE auth-readiness gate at the app shell is NOT covered by it and must implement its own latch. ### Dashboard sidebars are SHARED per nav level — one edit covers every page at that level

The left-nav is rendered by `apps/dashboard/src/components/context-sidebar.tsx` → `ContextSidebar`, which switches on the URL to ONE sidebar component per navigation level. There is no per-page sidebar. So a sidebar change requested on "the X page" almost always applies to **every page at that level**, and usually has an **analog at the campaign level** too. Before scoping any sidebar tweak (section header, link, badge, gate), map all levels and surface the full blast radius in the plan — don't scope to the single URL the user pasted.

| Level | Component | File | Group structure |
|-------|-----------|------|-----------------|
| App / Org | `AppLevelSidebar` / `OrgLevelSidebar` | `context-sidebar.tsx` | — |
| Brand | `BrandLevelSidebar` | `context-sidebar.tsx` | Overview · **Outcomes** · Features · Brand Settings |
| Feature (all feature pages) | `FeatureLevelSidebar` | `context-sidebar.tsx` | Campaigns · **Outcomes** (entities) · **Settings** (Workflows/Settings) · **Report** |
| Campaign (all campaign pages) | `CampaignSidebar` → `McpSidebar` | `campaign-sidebar.tsx` + `mcp-sidebar.tsx` | Overview · **Outcomes** (`outcomesItems`) · **Settings** (`settingsItems`, no Report) |

`McpSidebar`'s sole consumer is `CampaignSidebar` — edit it for campaign-level changes; it does not affect the feature/brand sidebars (those are hand-rolled in `context-sidebar.tsx`). The shared section-header style is `<h4 className="px-3 pb-1 text-xs font-semibold text-gray-400 uppercase tracking-wide">`. (Incident 2026-05-30 distribute.you#1223: section-header request pasted for one feature URL actually spanned every feature page + the campaign sidebar analog; only caught because the user pointed it out — plan v1 had scoped to the single page.)

**Deleting a page route ⇒ delete its nav scaffolding in the SAME PR, or the link 404s.** `context-sidebar.tsx` wires each nav level in THREE places: (1) the `SidebarItem` link (`href`), (2) the `NavigationLevel` type-union member + its `getNavigationLevel(segments)` branch, (3) the `XLevelSidebar` component + its `switch (level.type)` case. When you delete an `app/.../<segment>/page.tsx` route, grep `context-sidebar.tsx` for that segment and remove all three — a surviving link points at a route Next no longer serves and renders a hard 404 (NOT a route-segment `error.tsx`; that only catches render throws, not missing routes). The orphan is invisible to `tsc` (the link string is valid TS) and to the source-substring sidebar tests unless you assert its absence. Incident 2026-05-30 (distribute.you#1226): feature-level "Settings" link survived #877's deletion of `features/[featureSlug]/settings/page.tsx` (route-group migration + dynasty removal) along with its `featureSettings` nav level + `FeatureSettingsLevelSidebar`; the dead link 404'd for months until a user hit it. Fix removed all three + folded the surviving Workflows link into the feature `topItems`.

**Entity-count badge MUST mirror its page's own data source (`listingFallback`), not a feature-stat — a badge≠page divergence is a DASHBOARD bug even when the empty page is caused by an upstream data gap.** Both sidebars resolve an entity badge in this order: `listingFallback[entity.name]` (the count the page actually renders) → else `fStats[entity.countKey]` (features-service stat). The `listingFallback` map is the single-source mechanism: every entity whose page lists items MUST have an entry sourced from the SAME endpoint the page calls. An entity missing from `listingFallback` silently falls through to its `countKey` feature-stat, which can count a DIFFERENT data layer than the page shows. When two surfaces of THIS app disagree on the "same" entity's count, that is THIS repo's bug to fix (make them coherent) — do NOT linear-defer it to the backend just because the page's emptiness traces to an upstream gap; the upstream gap is a separate issue. When an entity has two views (feature-level + campaign-level), unify BOTH the page render AND both badges on ONE canonical endpoint — never mix layers (e.g. silver `listQuoteRequests` on one + gold `GET /orgs/opportunities` on the other). Incident 2026-05-30 (distribute.you#1231): `quote-requests` (feature `pr-expert-quote-opportunities`) was the one entity absent from both `listingFallback` maps → badge fell through to `quoteRequestsFound` (silver pool `provider_quote_requests` = 142) while the page rendered the gold scored catalog (`GET /orgs/opportunities` = 0, scoring never ran). Fixed by pointing the feature page + both badges at `listRankedOpportunities` (shared queryKey `["rankedOpportunities", {brandId}]`) so badge == page (0/0 now, N/N once scoring runs). First diagnosis wrongly led with "not a dashboard bug, linear-defer"; user corrected — coherence between the app's own surfaces is always the app's responsibility.

### Dynasty-First Display Rule (Workflows Only)

Always display `dynastyName` for workflows, never the versioned name. The only exception is settings/debug panels where the specific version matters — there, show the version number and versioned name alongside the dynasty name. This applies to page titles, table rows, cards, breadcrumbs, and any user-facing text.

Note: Features no longer have dynasty concepts. Features use `slug` and `name` directly.
