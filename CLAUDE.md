# Project: MCP Factory

DFY (Done-For-You), BYOK (Bring Your Own Keys) automation platform on MCP. User gives URL + budget; the platform handles lead finding, outreach, email generation, reporting.

## Commands

```bash
pnpm dev                    # All services via Turbo
pnpm dev:dashboard          # Dashboard only (Next.js, port 3001)
pnpm dev:<service-name>     # Any individual service
pnpm build                  # Build all (Turbo-orchestrated)
pnpm lint                   # Lint all packages
pnpm generate:readme        # Regenerate README.md from shared/content

# Per-package testing — scripts use `test`/`test:unit`, NOT `vitest` directly
pnpm --filter @distribute/<package> test                              # all tests
pnpm --filter @distribute/<package> test:unit                         # unit only
pnpm --filter @distribute/<package> test tests/unit/specific.test.ts  # single file
```

**Verify frontend with `tsc --noEmit`, NOT lint — `next lint` is GONE in Next 16.** Any `next lint` invocation reads `lint` as a directory arg and fails; no standalone `eslint` binary in the repo. Real check: `cd apps/<app> && node_modules/.bin/tsc --noEmit -p tsconfig.json` (dashboard has ~3 pre-existing test-file `TS1501` regex errors — ignore, only care about your files). (#1223)

**Don't trust RTK's `next build` summary** — the hook replaces stdout with a `Errors: N | Warnings: M` one-liner that can be wrong (phantom `Errors: 1` on a clean run). Real run: `rtk proxy -- npx next build` (the `--` is required), trust the log + exit code.

**Build shared workspace packages before app tests/build.** Vitest/Vite resolve workspace deps via `dist/`; an unbuilt `shared/*` surfaces as `Failed to resolve entry for package "@distribute/<name>"`. Run `pnpm -r build` after install or after pulling `shared/` changes.

**Source-substring page tests: a `not.toMatch(/X/)` guard trips on the fix's own explanatory comment** (a comment that writes the forbidden literal fails the test). Reword the comment, or assert against a comment-stripped copy (`pr-expert-public-report.test.ts` ships `stripComments(src)`). (#1235, DIS-112)

**Run `pnpm --filter @distribute/dashboard test` before pushing a dashboard change — `tsc` alone misses source-substring guards** (many tests assert a page contains a literal expression/copy string). `tsc` stays green while the guard goes red in CI and silently blocks auto-merge. Run it (~3s, source-only) and update the matching guard in the same commit. (#1252)

## `apps/admin` is the STAFF GOD-MODE console (admin.distribute.you) — a dashboard fork, NOT a customer surface

`apps/admin` is the dashboard cloned for internal staff. Two load-bearing facts before touching it:

This is so a staff tab reads distinct from a customer tab at a glance. A "swap the mark across all apps" PR (like #2614 which green-ified everything) MUST skip admin or re-clobbers this — keep the customer dashboard green, admin white/blue. The green `text-brand-600` "distribute" wordmark text was left as-is (only the mark differs). (#2625)
- **Access = email allowlist, single source `apps/admin/src/lib/admin-allowlist.ts`** (`ADMIN_ALLOWED_EMAILS` + `isAdminEmail`). Edge-gated in `proxy.ts` off the session `email` claim (403 for any signed-in non-staff on every non-public route); the `/api/admin/*` routes re-check it (defense-in-depth). Add/remove staff in that ONE file — don't re-inline the list. **The api-service `requireStaff` gate (staff-only `/v1/billing/credits/*` grant proxies + `/v1/billing/usage-discount` + admin-brands/features-admin, on the forwarded `x-email`) HARDCODES the same allowlist in `src/middleware/auth.ts` — the `STAFF_EMAILS` Railway env var was REMOVED (api-service v0.84.2, #698) because Kevin doesn't want a Railway var for this + it silently drifted. Whenever you add/remove staff here, edit the hardcoded array in api-service `src/middleware/auth.ts` to the SAME set + ship it, or the missing staff get a 403 on credit-granting / discount-setting. Do NOT re-introduce a `STAFF_EMAILS` env read.** (admin free-credits #2059; hardcode #698)
- **Cross-org "god-mode" = option A (real auto-join), NOT read-only impersonation.** The breadcrumb org switcher lists EVERY platform org via `GET /api/admin/orgs?q=` (Clerk Backend `getOrganizationList`, `CLERK_SECRET_KEY`), not `useOrganizationList({ userMemberships })`. On switch, `handleOrgSwitch` first `POST /api/admin/orgs/:orgId/join` (makes the admin a real `org:admin` member, idempotent) THEN `setActive` — because Clerk `setActive` rejects a non-member org. This keeps the whole **DIS-143 membership-scoped data path unchanged** (Clerk active org → JWT → gateway `x-external-org-id`); brands/features/writes all follow the active org with zero extra change. Accepted tradeoff: staff emails appear in each customer's Clerk member list. Why A over a read-only `x-admin-view-org-id` impersonation header: the gateway admin path (`api-service auth.ts`) trusts `x-external-org-id` with no membership check BUT resolves the (org,user) PAIR via client-service `/internal/resolve`, which **upserts the caller's `users.orgId` to the viewed org** (single org per user) — so a naive "forward URL org + staff user id" would churn/mis-attribute the staff user across customer orgs. Real membership (A) makes that churn legitimate + lets staff act, which is what's wanted. (#1643)
- **The SAME god-mode also lives in the customer `apps/dashboard`, but STAFF-GATED — and there the gating model flips.** `apps/dashboard/src/lib/admin-allowlist.ts` (byte-equal copy) + the same `GET /api/admin/orgs` + `POST /api/admin/orgs/:orgId/join` routes; breadcrumb shows the all-orgs switcher only when `isAdminEmail(useUser().email)` (else the unchanged `userMemberships` list — customers see only their own orgs). CRITICAL DIFFERENCE vs the admin app: the dashboard is the customer surface with **NO edge allowlist**, so the route-level `isAdminEmail` **403** is the SOLE security boundary — it's what stops a logged-in customer from enumerating/joining every other tenant. The client `isStaff` flag is cosmetic (hides the UI); never rely on it for access. Keep the dashboard allowlist + routes byte-equal with admin's. (#1655)
- **The god-mode join route MUST be idempotent via Clerk's RESPONSE, never via a `getOrganizationMembershipList` pre-check — that pre-check reads only the FIRST PAGE and silently breaks for staff who joined many orgs.** Symptom: god-mode switch into a (real, onboarded) customer org lands on a **blank org page** (`/orgs/:id` renders, sidebar + header fine, main empty). Causal chain (took a full session to trace, documented so it's instant next time): `POST /api/admin/orgs/:id/join` 502s → `OrgActivator` (dashboard, the URL-driven god-mode variant) does `if (!res.ok) throw` BEFORE `setActive`, so the active org never switches → URL org ≠ Clerk active org → `useAuthQuery`'s org-consistency gate (`urlOrg === activeOrg`) DISABLES every org-scoped read → the org-landing page's `["brands"]` query stays `undefined` → `if (!brandsData) return null` → blank; meanwhile every `/api/v1/*` returns **409 org_desync** (proxy `checkProxyOrg`: client-sent URL org ≠ JWT active org). The 502 root cause: the join route did `getOrganizationMembershipList({userId})` (FIRST PAGE only) → for a staff member of many orgs an existing membership beyond page 1 read as absent → re-`createOrganizationMembership` → Clerk **400 `already_a_member_in_organization`** → route wrapped it 502. **Tell it apart from the legit case:** an org the staff is NOT meant to enter (incomplete/never-onboarded org, no brand) bouncing to `/onboarding` is CORRECT (that's the first-run gate, not this bug) — THIS bug is a blank page on a COMPLETE org (has brands). **Fix (KevinLourd/godmode-join-idempotent):** drop the paged pre-check; `createOrganizationMembership` directly and treat a Clerk error whose `errors[].code === "already_a_member_in_organization"` as success (`{ok:true}`) — trust Clerk (server-side source of truth) over a paged list. Applied byte-equal to BOTH `apps/dashboard` + `apps/admin` join routes. Hardening in the same PR: `OrgActivator` now `await session?.getToken({skipCache:true})` AFTER `setActive` (mirrors `handleOrgSwitch`, CLAUDE.md DIS-143 item 3) to close the stale-token 409 window. **Diagnostic approach that nailed it:** the route only returned `err.message` ("Bad Request", Clerk's generic) — the real `errors[].code` was in the **Vercel runtime logs** (`get_runtime_logs query:"admin-join"` → `already_a_member_in_organization`); and the "is this org actually complete?" question was answered by querying **brand-service** (`org_brands` for the org → 2 brands = onboarded → blank is a bug, not the onboarding gate). Org NAMES live only in Clerk (client-service `orgs.name` is null for everyone), and the prod `CLERK_SECRET_KEY` Backend API is IP-restricted from a dev machine (403 code 1010) — so map a Clerk-org-name to data via brand-service, not Clerk. (godmode-join-idempotent)

## Currency display = adaptive by magnitude (`<$10` → cents, `≥$10` → whole), EXCEPT billing

Every USD amount on the **user dashboard** formats by magnitude: `abs(usd) < 10` → 2 decimals (`$4.27`), `≥ $10` → 0 decimals rounded, thousand-sep (`$1,240`). Canonical helpers `formatUsdAdaptive(usd)` / `formatCentsAsUsdAdaptive(cents)` in `lib/format-number.ts`; `lib/format-stat.ts` currency branch + every live per-component currency formatter (revenue cards, strategy CPC/cost tiles, audiences, brand-status unit-cost, brand-info) implement the same `Math.abs(usd) < 10 ? 2 : 0` inline. **Billing is deliberately EXEMPT** — `billing/page.tsx` + `lib/billing-guard.tsx` keep the explicit-decimal `formatUsd`/`formatCentsAsUsd(…,2|0)`/`formatBillingCents` (exact charged amounts must never round). **DAILY BUDGET is a further carve-out — it ALWAYS renders as WHOLE dollars (no cents at all, not even under $10), a daily budget being an exact whole-dollar config value where cents read wrong (`$7.50/day` → `$8/day`).** Whole-budget surfaces: `brand-status-control` (`fmtUsdWhole` — status pill `budgetLabel` + launch budget tiers + "Daily budget: $X / day"; the unit-cost-per-outcome beside it stays adaptive), `revenue-cost-summary` (`formatBudgetCents` — the `/$budget` denominator), `billing-guard` (the "Brand daily budget cap / day" line only; its "Required" credit amount stays exact), onboarding (already whole via `formatLocaleInteger`), settings budget editor (`step=1` integer input). ## Release flow (distribute.you specifics)

No `release.sh hotfix` here (that targets Railway semver services). Vercel deploys on every `main` merge — no tag, no bump.

**NEVER run `vercel --prod` or `vercel deploy --prod` locally.** This bypasses git entirely, deploys a local build directly to prod, and wipes the Vercel project settings (`rootDirectory`, `framework`, `buildCommand` → all `null`). This has caused 3+ prod outages (pricing 404, stuck alias, copy regressions). The guard hook blocks it. If you see a `2–4s` "build" by `adam-atomic-git` in `vercel ls`, that's a local deploy — run the recovery below. Always push to `origin` via PR and let Vercel auto-deploy from the merge commit.

**Conductor workflow — create a new workspace per task, never `git checkout -b` manually.** Conductor creates each workspace as a git worktree on its own branch and fetches `origin` first, so the branch already starts from the latest remote commit. Branch prefix: `adam/<workspace-name>` (configured via `.conductor/settings.local.toml`). If the workspace has been open a while and `origin/main` has advanced, run `git fetch origin main` before creating a PR.

**Before REBUILDING an existing dashboard/admin PAGE, `git fetch origin main && git log origin/main -20 -- <page dir>` FIRST — the workspace branch lags `origin/main`, and a PARALLEL rebuild of the SAME page (or its backend contract) may have already merged.** The in-flight-workspace + `gh pr list --search` dup-checks are NOT enough: they don't surface a page rebuild that already landed on THIS repo's `origin/main`. Cost 2026-07-05 (admin CRM contacts rebuild): built the entire page + spawned a 4-repo backend fan-out against the OLD server-fetch CRM page, but `origin/main` had already merged a client-side rebuild (#2391 — `GoogleCrmClient` + typed safeParse readers) AND the backend contract landed in parallel autonomous workspaces (google-service silver + `contact-links` + `?participant`, brand-service `/internal/brands/all`, api-service proxies). Had to `git reset --hard origin/main` and rebuild the feature (contacts table + right-panel status editor) ON TOP of the merged base, adopting its readers. Two corollaries: (a) adopt the merged base's typed readers/types and EXTEND them (add `links`/`participant`/`saveContactLinks`), never re-declare a parallel set; (b) an "overlapping" in-flight workspace editing the SAME files is NOT necessarily a conflict — additive changes that merge sequentially are fine (the silver + contact-links tasks both edited `orgs-google.ts` and both merged clean), so don't alarm/discard a sibling workspace prematurely.

- **Bugfix / Feature** → branch from `origin/staging`, PR → `staging`. Promotion to `main` via the staging→main PR flow.
- **Exception (`apps/dashboard/` + `apps/landing/` only):** dashboard-only and landing-only PRs merge straight to `main` in recent practice.   - **Uncommitted working-tree edits AUTO-FOLLOW `git checkout -b NEW origin/main` — do NOT then `git checkout - -- <path>` to "carry" them.** `checkout -b` keeps your dirty edits on the new branch already. Running `git checkout - -- <path>` afterward restores that file from the PREVIOUS branch's committed HEAD, silently REVERTING your uncommitted edit (the Edit tool's in-memory file-state still thinks it's applied → a re-Edit fails "File has not been read"; re-Read shows the old code). After `checkout -b`, just `git diff --stat` to confirm the edit rode along, then commit — never re-checkout the file. (#1966: lost the edit this way, had to re-Read + re-apply.)
- **Chain `git commit && git push` with `&&`, never `;`/newline — Conductor worktrees hit `index.lock` contention.** A concurrent git process (other agent, IDE) can make `git commit` fail with `Unable to create '.git/worktrees/<ws>/index.lock'`; if the next line is an unconditional `git push`, it pushes the **un-advanced HEAD** → an empty branch at `origin/main`, and the `* [new branch]` output reads like success. Always `git commit … && git push …` so the push only fires on a committed HEAD; if you hit the lock, `rm -f .git/worktrees/<ws>/index.lock`, re-commit, verify `git log --oneline -1` advanced, then push. (#1367)

**A brand's "status" (active / paused / inactive) is a COMPOSITE of TWO services — pause + budget — never one.** The `paused` flag lives in **campaign-service** (`brand_pause` table `{brandId, orgId, paused}`, read via `GET /brands/:brandId/pause`, api-service `brand-pause.ts` forwards there — explicitly *NOT* brand-service, *NOT* billing). ### Verify backend shape before writing client types (wire-shape rot)

**Deployed openapi is ground truth. Before declaring ANY typed helper (`listX`/`getX`/response interface/Zod schema) for a backend endpoint, fetch the real shape via the `api-registry` / `api-registry-staging` MCP.** Never invent from "what feels right" or an aspirational PR description. **Live (MCP `call_api`) > source > local clone** — local clones (incl. a subagent's) go stale; verify a subagent's reported wire shape via the live MCP. Want a richer shape → file a backend request and block. Long-lived types rot silently when an upstream endpoint is refactored — re-check periodically, not just at creation. (#1079, #1094)

**A backend-contract migration on `apps/dashboard/src/lib/api.ts` MUST migrate the byte-equal reader in `apps/admin/src/lib/api.ts` in LOCKSTEP — the two apps share the same `getX`/`listX` Zod readers and DRIFT.** admin is a dashboard fork with its own copy of `api.ts`; a reshape fixed in only one app leaves the other's `safeParse` throwing on every consumer of that endpoint (loud console `response shape mismatch` + broken page). This has bitten in BOTH directions: dashboard-straggler on the ahref DR/traffic bulk readers (#2070), and admin-straggler on the `getWorkflowProjection` `{workflows[]}`→`{rows[]}` ladder reshape (dashboard fixed #2445/#2446, admin still legacy → #2451). When you migrate a reader for a reshaped/renamed backend contract, `grep` the SAME symbol in the sibling app's `api.ts` and port it in the same PR (or spawn the sibling straggler); prefer keeping the two readers byte-equal, diverging only where a support type is genuinely absent in one app. (#2451)

- **A bulk reader keyed by a `?domains=a.com,b.com,…` (or `?ids=`) query string FAILS SILENTLY at scale — the URL/header size limit (~8–16KB) is blown long before you notice, the request errors, and the consumer's map comes back EMPTY (blank column/badge, no crash).** A brand can own 12k+ outlet domains; `getDomainDrStatuses` / `getDomainTrafficHistories` passed every domain in ONE GET (~180KB URL) → both the CSV DR/Monthly-Visits columns AND the cards went blank. ### Local-first SWR cache (per-query IndexedDB persister) — open a page → its content NOW

**This is the dashboard's named architecture: a LOCAL-FIRST, stale-while-revalidate (SWR) cache.** Local-first = the on-disk cache is the source the UI paints FIRST, network is secondary. SWR = show the cached (stale) data instantly, refetch in the background, swap fresh in silently (TkDodo: *"stale data is better than no data, because no data means a loading spinner = perceived slow"*). It is the 4th "don't revert resolved state on a transient" layer and the only one that survives cache EVICTION/RELOAD (the first three keep warm-in-memory state across refetch only): 1. `keepPreviousData` (data) · 2. `useCoordinatedReveal` (reveal) · 3. `useMonotonicStatuses` (row bucket).

**Mechanism (`src/lib/query-provider.tsx` + `src/lib/persist-cache.ts`, unit-tested):** the **PER-QUERY persister** `experimental_createQueryPersister` (from `@tanstack/react-query-persist-client`), wired as a default query option (`queries.persister = persister.persisterFn`) on a plain `QueryClientProvider`. Storage = **IndexedDB** via `idb-keyval`. NOT the old whole-client `PersistQueryClientProvider` + `createSyncStoragePersister` (localStorage).

**Why the migration (2026-06-25, KevinLourd/dashboard-instant-cache) — the 2026-06-23 "persist-everything into localStorage" policy was SELF-DEFEATING, causing the exact two symptoms it meant to fix ("30s skeletons all the time" + "site lourd/lent"):**
- **localStorage's hard ~5 MB per-origin cap** → a heavy brand's `brandLeads`/`brandEmails` blew the cap → `removeOldestQuery` evicted the small overview queries (cached first) → the overview cold-skeletoned on the slow Neon chain. The persist-all policy evicted the very page it was meant to keep.
- **Whole-client `dehydrate()` re-serializes the ENTIRE cache on EVERY mutation** (TanStack #9775). ~12 overview queries polling 5s → constant multi-MB `JSON.stringify` + sync write on the MAIN THREAD → jank/"lourd".
- The per-query persister fixes BOTH: each query is written **separately, keyed by its query hash, only when IT changes** (no whole-cache re-serialize) AND IndexedDB has **no 5 MB cap + async off-main-thread writes**. Verified against TanStack `createPersister` docs + TkDodo #5169.

**`admin` ≠ `dashboard`: separate Vercel apps = separate ORIGINS = separate storage.** `admin.distribute.you`'s heavy Outlets/Journalists/Articles cache NEVER touches the `dashboard.distribute.you` cache. Don't diagnose dashboard slowness via admin's big lists; the dashboard customer never even loads those (alpha/staff-gated). Admin is allowed to stay heavy.

**Load-bearing rules:**
- **`maxAge: Infinity` is a LIE without `navigator.storage.persist()` — request persistent storage at boot, or the browser EVICTS the whole IndexedDB store after a few idle days and every `useAuthQuery` surface cold-skeletons again.** Best-effort IndexedDB (the default) is evictable under disk pressure / prolonged non-use, so the "instant across days" the per-query persister promises silently decays — this is the recurring "instant right after a deploy, infinite skeleton a few days later" report on audiences/leads. `QueryProvider` calls `requestPersistentStorage()` once at mount (feature-detected, idempotent via `storage.persisted()`, never throws). Do NOT remove it; a rebrand/refactor of `query-provider.tsx` must keep the boot call. Pairs with the org-gate latch below — eviction UNMASKS the disabled-query hang. (KevinLourd/leads-skeleton-persist-orggate)
- **`maxAge: Infinity` (disk = keep forever, never expire) + `gcTime: 30min` (memory only) are DECOUPLED.** This decoupling is the whole reason per-query beats whole-client: with the per-query persister a heap-GC'd query **stays on disk**, so "instant on open, any session, any day" needs NO `gcTime: Infinity` → no heap blowup. (The whole-client persister mirrors disk⇆memory, so `maxAge: Infinity` there would force `gcTime: Infinity` = the #1273 OOM — impossible to have both there.) `PERSIST_MAX_AGE_MS = Infinity`, `PERSIST_GC_TIME_MS = 30min`. The old "`gcTime` MUST equal `maxAge`" rule (TanStack #5169) applies ONLY to the whole-client persister — it does NOT bind here.
- **Org-scope via the persister `prefix`** = `persisterStorageKey(orgId)` (`distribute-dashboard-cache:{orgId}`); each query's storage key is `${prefix}-${queryHash}`, so org A's keys never collide with org B (DIS-143). **`storage: undefined` while `orgId` is null OR on the server** (`persistEnabled = typeof window !== "undefined" && !!orgId`) — no anon-bucket cross-org bleed. The keyed remount (`key={orgKey}`) still resets the in-memory cache on switch.
- **`buster` = MANUAL version constant** (`persistCacheVersion()` → `PERSIST_CACHE_VERSION`, currently `"1"`), bumped BY HAND only on an incompatible shape change — NEVER the git commit SHA (the SHA flips every deploy, ~12×/day → wiped the disk cache on ~every visit, defeating persistence). With `maxAge: Infinity` there's no expiry safety net, so `buster` is the ONLY forced invalidation; cross-deploy shape drift is otherwise tolerated by `safeParse`/`z.coerce` on list readers + keep-last-good `structuralSharing`. Bump when you rename/remove a field on a response type consumed straight from cache with no safeParse guard (additive optional fields do NOT need a bump).
- **Persist filter = success + non-sensitive + ALLOWLISTED** via the persister `filters.predicate` → `shouldPersistQuery` (reused from the old `shouldDehydrateQuery`). `PERSISTABLE_QUERY_ROOTS` is the INVENTORY of every live non-sensitive root (big lists included); default-OFF for an unlisted root. Secrets EXCLUDED (`apiKeys`/`byokKeys`/`keySources` via `SENSITIVE_QUERY_ROOTS` — key material never on disk).

**Irreducible:** the genuine FIRST-EVER load on an empty cache still waits on the cold Neon backend (~20-30s, features/api/runs scale-to-zero 0.25 CU). Killing that needs backend warm-up (other repos), NOT a dashboard cache change.

**Backend-DOWN must degrade to STALE, never to a skeleton — and the persister has TWO disk-restore paths, each with a hole that a down backend un-masks.** The per-query persister self-restores a query from disk ONLY when that query actually FETCHES (i.e. `enabled`); the org-consistency gate (`use-auth-query.ts`) disables every `useAuthQuery` while Clerk's active-org is still settling, so during that window the disabled queries never fetch → never self-restore. The other path — the mount seed `persister.restoreQueries` (query-provider.tsx) — is ONE-SHOT per provider mount. So a page reached while the gate is momentarily closed, OR a sub-page whose memory was GC'd (30-min `gcTime`), cold-skeletons even though its stale snapshot sits on disk. Backend-HEALTHY hides this (the network answers within seconds); backend-DOWN turns the transient into a STUCK multi-minute skeleton on a page the user opens 10×/day — the local-first cache silently not painting. Diagnose by reading the disk directly in the browser console: `(await indexedDB.databases())` + a `keyval-store` getAllKeys — if `distribute-dashboard-cache:<org>-["<root>",…]` entries EXIST for the affected org/brand while the page skeletons, the data is on disk and NOT being painted (a restore-path bug, NOT eviction; eviction = the store/entries are absent, a different failure fixed by `requestPersistentStorage` #2336). **Fix (#2574): `reseedColdQueriesFromDisk` re-seeds COLD queries from this org's on-disk snapshot on EVERY org-scoped navigation, decoupled from `enabled`/network** — pure guarded helper `coldRestorablePairs` (org-prefixed = DIS-143, buster-checked = never paint stale-shaped data, COLD-GUARDED = `getQueryData!==undefined` skip so a reseed never stomps fresher live memory), read-only `setQueryData`, zero network. Any new org-scoped restore/reveal work must keep the invariant: painting a present disk snapshot is decoupled from the gate + the network, so a down backend always shows stale. (#2574, sibling of the #2336/#2337 eviction+org-gate fix — distinct: there the disk was empty, here it's present-but-unpainted.)

**Rule:** persistence is GLOBAL — one per-query persister wired as a default query option, never a second provider / per-page persister. `experimental_createQueryPersister` is `experimental_`-prefixed (stable behavior, API may shift — pin to the installed `@tanstack/react-query-persist-client` version). Keep `PERSISTABLE_QUERY_ROOTS` in lockstep with live queries. (#1273 small-allowlist origin; #1278/DIS-170 30min+allowlist; 2026-06-23 persist-all-localStorage regression; 2026-06-25 KevinLourd/dashboard-instant-cache = per-query IndexedDB + maxAge Infinity, the current design.)

### keep-last-good (`structuralSharing`, cache-write boundary) — a SUCCESSFUL refetch returning VALID-but-degenerate data must not collapse the UI

5th "don't revert resolved state on a transient" layer — and the ONLY one for a **successful** refetch that legitimately returns **worse** data. The other four all assume the new data is good: 1. `keepPreviousData` (key-change / error→undefined) · 2. `useCoordinatedReveal` (reveal decision) · 3. `useMonotonicStatuses` (status downgrade *ordering* on a poll) · 4. persisted cache (eviction). **The gap:** a 200 whose nullable field flips non-null→null (a cold Neon chain — api→features→workflow/runs/email-gateway/brand, all scale-to-zero — answers a poll/refocus refetch with zeroed unit costs or fewer rows while half-warming). `keepPreviousData` can NOT help — the null IS the new valid query data, so React Query installs it and any UI **derived** off it (budget cards, a `disabled` gate, a badge, a $ amount) collapses mid-session.

**Industry-aligned fix (researched: TkDodo / TanStack docs / SWR all agree there is NO built-in for successful-but-worse — you intercept at the cache-write boundary).** TanStack's hook is `structuralSharing(prev, next)` (SWR's is `compare`, Apollo's is a `merge` field policy). Helpers in `src/lib/keep-last-good.ts`: `keepLastGoodFields(prev, next, fields, label)` (per-field coalesce: keep prev's non-null when next nulls it) + `keepLastGoodList(prev, next, {keyFn, fields, label})` (keyed merge: per-item coalesce + RETAIN a prev item that vanished from next). Both `console.error` the suppressed downgrade (fail-loud — a persistent real downgrade still logs; this is keep-resolved, NOT a silent fallback). Compose per query (e.g. `keepLastGoodWorkflowProjection` in `api.ts`, co-located with the type) and pass it as `structuralSharing` on the `useAuthQuery` — protects EVERY consumer of that query at ONE place, not per-component (a per-component latch is opt-in-and-forgettable, which is exactly how the collapse shipped).

**Five load-bearing rules:**
- **OPT-IN per query, NEVER a global default.** A null can legitimately mean "deleted/gone"; blanket last-good would mask real deletions. Apply ONLY where a null means "transient/not-ready" (cold compute, in-flight aggregation), not "removed".
- **Put it at the QUERY (`structuralSharing`), not at each consumer.** One merge → cards + button + badges all inherit it. The per-component `useLastGoodValue` latch is strictly weaker (forgettable).
- **List ONLY the fields whose null = transient.** Fields outside the allowlist let next win (incl. when null) — so a genuinely-cleared field still clears.
- **CUT the *steady-state* poll on econ-INDEPENDENT slow-changing queries — but keep a BOUNDED cold-start warm-up poll.** The workflow-projection unit costs are "fetched ONCE" by design (#1528/#1536) — a perpetual 5s `...pollOptions` on them only re-hits the cold Neon chain for zero steady-state benefit (cards update from the client `econLive` recompute). BUT "fetch-once + `refetchOnWindowFocus`" alone is NOT correct: **keep-last-good cannot heal the FIRST fetch** (no prior good value to keep), and the cold Neon chain most often returns its degenerate-200 (null unit costs / empty workflows) on exactly that first request — so a cold load STICKS on the empty-state ("No workflow has website-visit cost data yet") until a manual window-refocus. The right shape is a **conditional `refetchInterval`**: poll (~4s) ONLY while the payload has no usable data (e.g. no workflow with non-null `costPerCloseUsd`), capped at N tries (so a genuinely-empty brand stops), `false` the instant it warms. This self-heals cold-start without perpetual polling. (#1557-regression: #1542 dropped the poll entirely calling it "pointless"; it was the cold-start self-heal — keep-last-good #1548 only covers a later bad refetch of an already-good cache, never the first fetch.)
- **TanStack types `structuralSharing` params as `unknown`** — pass `(prev, next) => keepLastGoodX(prev as T | undefined, next as T)`.

**Rule:** any **polled / refetch-prone** query with **nullable fields that GATE or compute UI** (disabled-button, empty-state, $ amount, recommended pick) MUST pass a keep-last-good `structuralSharing`. Pure read-display surfaces that already degrade to "—" on null are lower-risk (phased-sweep, not a hotfix). (#1542-regression: the launch-page budget cards + Launch button — then `campaigns/new`, now `brands/[brandId]/launch` — collapsed when the polled `workflowProjection` caught a valid-but-degenerate 200 mid-session; `econLive`/§2 inputs were NOT the cause — they're latched. The poll predated #1542; #1542 widened it by recomputing cpc client-side. Fix: keep-last-good `structuralSharing` on the query + drop the pointless 5s poll. Other projection/revenue surfaces — conversions/workflows/brand-overview/report + top-workflows card — are the phased follow-up; the campaign-LEVEL variants (`campaigns/[id]/overview`, top-campaigns card) were removed in #1768.)

### Org switching — cross-org isolation framework (defense-in-depth)

4th "don't act under the wrong context" member (after data/reveal/bucket). Keeps every API call + the whole cache bound to the viewed org. **Failure:** the proxy forwards the Clerk session active org (`auth().orgId`), decoupled from the URL `/orgs/[orgId]` (Clerk `organizationSyncOptions` syncs URL→session on page routes, NOT `/api/*`); after `setActive` the JWT rotates async → a lag window carrying the OLD org → a stale write commits under the old org (later 404) or a stale poll 404s. campaign-service isolation is correct; the 404 is the symptom. (DIS-143). Doctrine converges (Clerk + OWASP Multi-Tenant + tkdodo): derive tenant from JWT claims not client headers, validate every request, fail CLOSED, remount on switch.

Framework (every layer central, one file — never per-page):
1. **Server choke point — fail-closed proxy guard** (`src/lib/proxy-org.ts` `checkProxyOrg`): the client sends `x-active-org-id` (the URL org); the proxy compares it to `auth().orgId`, returns **409 `org_desync`** on mismatch, refuses to forward. The JWT stays the sole authority (client value detects the desync only — OWASP fail-closed, not trust-the-header). Applied to ALL proxy routes; a new route → call `checkProxyOrg` right after the `clerkOrgId` check. Constants in dependency-free `src/lib/org-desync.ts`.
2. **Client cache choke point — keyed remount** (`src/lib/query-provider.tsx`): inner `QueryClientProvider key={org.id}` → org switch = new client = empty cache atomically. REPLACES `queryClient.clear()` (raced). So `OrgCacheInvalidator` MUST mount ABOVE `QueryProvider` (survives the remount); it no longer touches the cache NOR navigates — it ONLY clears breadcrumb caches on an org change (the cross-tab `router.push` was removed in #1944, see item 5). **The remount key MUST be MONOTONIC — latch it to the last RESOLVED org id, never raw `orgId ?? "no-org"`.** Clerk's `useOrganization()` blinks `organization: null` transiently during background JWT rotation (~1/min) + tab focus/reconnect; a raw key flips `realId→"no-org"→realId` on every blink → spurious remount → empty in-memory cache. Persisted queries rehydrate from IndexedDB silently (ALL live roots, big lists included, no 5MB cap since the 2026-06-25 per-query IndexedDB migration — so this blink-skeleton is gone). Fix: `const lastOrgId = useRef(null); if (orgId) lastOrgId.current = orgId;` then key + `orgId` prop both off `lastOrgId.current` — a null blink keeps the last key (no remount); a real switch to a DIFFERENT org still advances the id → remount + fresh per-org persister prefix, DIS-143 intact. Same monotonic-gate rule as the app-shell auth latch. (#1767)
3. **Source — `await setActive` THEN `await session?.getToken({ skipCache: true })`** (`breadcrumb-nav.tsx` `handleOrgSwitch`) before `router.push`. `await setActive` alone is NOT enough: it resolves on the CLIENT before its `Set-Cookie` propagates, so the very next navigation reaches `proxy.ts` `organizationSyncOptions` carrying the STALE token (active = previous org / not-a-member of the target). Clerk bounces the URL back → `OrgActivator` re-syncs the client to the previous org → **the switch reverts on its own** (loudest in STAFF god-mode, where the target is a just-joined customer org whose membership the stale JWT lacks). The `getToken({ skipCache: true })` re-mint forces the fresh token (new active org + new membership) into the cookie BEFORE the middleware reads it — same proven fix as `beta-onboarding`'s onboarding-complete hop (CLAUDE.md "Stale token — force `getToken({ skipCache: true })` before navigating"). Guarded by `org-switch-cache.test.ts` (getToken must sit between setActive and router.push). (#1940)
4. **Read gate** (`src/lib/use-auth-query.ts`): `enabled: caller && urlOrg === activeOrg`. **LATCHED against Clerk's transient `organization: null` blink** (JWT rotation ~1/min, tab focus): the gate compares `urlOrgId` to `effectiveActiveOrgId = activeOrgId ?? lastResolvedActiveOrgId.current` (a `useRef` that only ever stores a NON-null active org). Without the latch a null blink flips the gate false → v5 disabled query → `isPending: true` FOREVER → any `isPending`-gated skeleton (leads page) hangs, and once the IndexedDB cache is evicted (see persist section) it hangs PERMANENTLY = the infinite-skeleton report. The latch fills ONLY a null — a genuinely DIFFERENT non-null active org (real switch / cross-tab) STILL closes the gate, so a null-from-cold never fires a read and DIS-143 cross-org isolation is intact (never fire a URL-org read under another org's token → proxy 409). NO `|| !activeOrgId` escape (a read under a truly-null active org lands in a null-org cache entry, TanStack #3743). `useOrganization` import allowed here (powers the gate, not token-passing). Guarded by `org-switch-cache.test.ts`. (KevinLourd/leads-skeleton-persist-orggate)
5. **MULTI-TAB — scope every request by the PER-TAB token, NEVER the shared cookie; never navigate a tab on the shared active-org signal.** Clerk supports a per-tab active org, BUT **the session cookie is one global singleton for the whole browser** — it reflects whichever tab was focused last ([Clerk: Organizations → "multiple browser tabs"](https://clerk.com/docs/guides/organizations/overview)). So scoping data off the cookie (`auth()` with no Bearer) makes a background poll / navigation from a NON-focused tab run under the WRONG org → cross-org bleed + 409 churn; and any code that NAVIGATES a tab when the (shared) `useOrganization()` active-org changes yanks tab A onto tab B's org the instant tab B switches — the "org switches by itself across tabs" bug. Two-part fix (#1944), per Clerk's own multi-tab guidance + WorkOS/SuperTokens "the token IS the tenant context":
   - **Data path:** `api.ts` `apiCall` attaches THIS tab's token — `Authorization: Bearer ${await window.Clerk.session.getToken()}` (via `getTabSessionToken()`, a non-hook read of the per-tab `window.Clerk` client) — on every `/api/v1/*` proxy call. ### Public marketing pages — SEO + AI-scraper rendering

For any public landing/marketing/docs page (anything under `apps/landing/`, `apps/sales-cold-emails-landing/`, `apps/docs/`), the default rendering strategy is **ISR + `unstable_cache` + edge cache**, NEVER Suspense streaming on indexable content.

Why: Next.js streaming SSR injects content into `<template>` blocks resolved by client-side JS. Googlebot (Chromium-headless) executes JS and sees the resolved content, but **AI scrapers (GPTBot, ClaudeBot, PerplexityBot, Bingbot AI, curl-like crawlers) parse raw HTML only** — they receive the skeleton fallback, not the data. Putting leaderboards, benchmarks, blog lists, stats behind `<Suspense>` kills AI Search indexing for the very content that earns AI Search traffic.

Concrete rules:

1. **No `<Suspense>` on indexable content.** Hero copy, h1/h2, body text, JSON-LD blocks, leaderboards, benchmark tables, blog cards, stats — all SSR-sync in the byte stream. Suspense reserved for user-specific, non-indexable widgets (logged-in dashboards, real-time counters, chat).
2. **Never combine `force-dynamic` with `revalidate`.** `force-dynamic` wins and ISR is silently disabled. Prefer ISR (`export const revalidate = N`) alone; let `unstable_cache` + graceful build-time empty-result handle the missing-config case.
3. **Calling `headers()` or `cookies()` in a page forces dynamic rendering** even with `revalidate` set. Drop them if the page can be ISR-prerendered (resolve hostname-dependent URLs from `NEXT_PUBLIC_*` env vars instead).
4. **Wrap every fetch / DB call used by a public page in `unstable_cache`** with explicit tags. Edge cache (`s-maxage / stale-while-revalidate`) provides instant HTML; `unstable_cache` provides instant data on the rebuild after invalidation. Both stack.
5. **Invalidate via `revalidateTag(tag, profile)`** in the publish/update flow (webhooks, admin forms). Next 16.2+ requires the 2nd arg `profile: 'default' | 'seconds' | 'minutes' | 'hours' | 'days' | 'weeks' | 'max' | CacheLifeConfig`. Pass `"default"` for ordinary invalidations.
6. **`unstable_cache` at module top level breaks vitest** because the wrapping runs at import time before `vi.doMock` can intercept. Mock `next/cache` globally via `apps/landing/tests/setup.ts` (`unstable_cache` → pass-through identity, `revalidatePath` / `revalidateTag` → `vi.fn()`); declare `setupFiles: ["tests/setup.ts"]` in `vitest.config.ts`.
7. **Validate scraper output after any caching change** with `apps/landing/scripts/seo-snapshot.sh [BASE_URL]` (counts `<h1>`, `<h2>`, `<article>`, JSON-LD blocks under a `GPTBot` user-agent) and `apps/landing/scripts/measure-ttfb.sh [BASE_URL]` (warm/cold TTFB per route). Both pre- and post-change snapshots in the PR description.

Incident 2026-05-24 (distribute.you#1153): pre-refactor, `/blog` TTFB was 1.5–2.3s because `force-dynamic` overrode `revalidate = 60` and every visit hit Neon directly. `/benchmarks` was dynamic because `headers()` was called for hostname resolution. `/investors` took 6–8s end-to-end because metrics were `cache: "no-store"`. Three loading.tsx files existed but were never reachable (page components weren't async-suspended). Single PR flipped 8+ routes from `ƒ Dynamic` to `○ Static` ISR with `unstable_cache` + tag invalidation; no Suspense or skeleton was added on any indexable content.
