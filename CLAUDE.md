# Project: MCP Factory

DFY (Done-For-You), BYOK (Bring Your Own Keys) automation platform on MCP. User gives URL + budget; the platform handles lead finding, outreach, email generation, reporting.


**Verify frontend with `tsc --noEmit`, NOT lint ŌĆö `next lint` is GONE in Next 16.** Any `next lint` invocation reads `lint` as a directory arg and fails; no standalone `eslint` binary in the repo. **Build shared workspace packages before app tests/build.** Vitest/Vite resolve workspace deps via `dist/`; an unbuilt `shared/*` surfaces as `Failed to resolve entry for package "@distribute/<name>"`. Run `pnpm -r build` after install or after pulling `shared/` changes.

**Source-substring page tests: a `not.toMatch(/X/)` guard trips on the fix's own explanatory comment** (a comment that writes the forbidden literal fails the test). Reword the comment, or assert against a comment-stripped copy (`pr-expert-public-report.test.ts` ships `stripComments(src)`). (#1235, DIS-112)

## `apps/admin` is the STAFF GOD-MODE console (admin.distribute.you) ŌĆö a dashboard fork, NOT a customer surface

`apps/admin` is the dashboard cloned for internal staff. Two load-bearing facts before touching it:
- **Access = email allowlist, single source `apps/admin/src/lib/admin-allowlist.ts`** (`ADMIN_ALLOWED_EMAILS` + `isAdminEmail`). Edge-gated in `proxy.ts` off the session `email` claim (403 for any signed-in non-staff on every non-public route); the `/api/admin/*` routes re-check it (defense-in-depth). Add/remove staff in that ONE file ŌĆö don't re-inline the list. **The api-service gateway `STAFF_EMAILS` env var (gates the staff-only `/v1/billing/credits/*` grant proxies via `requireStaff` on the forwarded `x-email`) MUST mirror this allowlist ŌĆö it is a THIRD copy that drifts: a code-only deploy can't keep it in sync, and a spawn told to "set STAFF_EMAILS" easily sets just one email. Whenever you add/remove staff here, set `STAFF_EMAILS` on api-service Railway (production) to the SAME comma-separated set, or the missing staff get a 403 on credit-granting.** (admin free-credits, #2059)
- **Cross-org "god-mode" = option A (real auto-join), NOT read-only impersonation.** The breadcrumb org switcher lists EVERY platform org via `GET /api/admin/orgs?q=` (Clerk Backend `getOrganizationList`, `CLERK_SECRET_KEY`), not `useOrganizationList({ userMemberships })`. On switch, `handleOrgSwitch` first `POST /api/admin/orgs/:orgId/join` (makes the admin a real `org:admin` member, idempotent) THEN `setActive` ŌĆö because Clerk `setActive` rejects a non-member org. This keeps the whole **DIS-143 membership-scoped data path unchanged** (Clerk active org ŌåÆ JWT ŌåÆ gateway `x-external-org-id`); brands/features/writes all follow the active org with zero extra change. Accepted tradeoff: staff emails appear in each customer's Clerk member list. Why A over a read-only `x-admin-view-org-id` impersonation header: the gateway admin path (`api-service auth.ts`) trusts `x-external-org-id` with no membership check BUT resolves the (org,user) PAIR via client-service `/internal/resolve`, which **upserts the caller's `users.orgId` to the viewed org** (single org per user) ŌĆö so a naive "forward URL org + staff user id" would churn/mis-attribute the staff user across customer orgs. Real membership (A) makes that churn legitimate + lets staff act, which is what's wanted. (#1643)
- **The SAME god-mode also lives in the customer `apps/dashboard`, but STAFF-GATED ŌĆö and there the gating model flips.** `apps/dashboard/src/lib/admin-allowlist.ts` (byte-equal copy) + the same `GET /api/admin/orgs` + `POST /api/admin/orgs/:orgId/join` routes; breadcrumb shows the all-orgs switcher only when `isAdminEmail(useUser().email)` (else the unchanged `userMemberships` list ŌĆö customers see only their own orgs). CRITICAL DIFFERENCE vs the admin app: the dashboard is the customer surface with **NO edge allowlist**, so the route-level `isAdminEmail` **403** is the SOLE security boundary ŌĆö it's what stops a logged-in customer from enumerating/joining every other tenant. The client `isStaff` flag is cosmetic (hides the UI); never rely on it for access. Keep the dashboard allowlist + routes byte-equal with admin's. (#1655)
- **The god-mode join route MUST be idempotent via Clerk's RESPONSE, never via a `getOrganizationMembershipList` pre-check ŌĆö that pre-check reads only the FIRST PAGE and silently breaks for staff who joined many orgs.** Symptom: god-mode switch into a (real, onboarded) customer org lands on a **blank org page** (`/orgs/:id` renders, sidebar + header fine, main empty). Causal chain (took a full session to trace, documented so it's instant next time): `POST /api/admin/orgs/:id/join` 502s ŌåÆ `OrgActivator` (dashboard, the URL-driven god-mode variant) does `if (!res.ok) throw` BEFORE `setActive`, so the active org never switches ŌåÆ URL org ŌēĀ Clerk active org ŌåÆ `useAuthQuery`'s org-consistency gate (`urlOrg === activeOrg`) DISABLES every org-scoped read ŌåÆ the org-landing page's `["brands"]` query stays `undefined` ŌåÆ `if (!brandsData) return null` ŌåÆ blank; meanwhile every `/api/v1/*` returns **409 org_desync** (proxy `checkProxyOrg`: client-sent URL org ŌēĀ JWT active org). The 502 root cause: the join route did `getOrganizationMembershipList({userId})` (FIRST PAGE only) ŌåÆ for a staff member of many orgs an existing membership beyond page 1 read as absent ŌåÆ re-`createOrganizationMembership` ŌåÆ Clerk **400 `already_a_member_in_organization`** ŌåÆ route wrapped it 502. **Tell it apart from the legit case:** an org the staff is NOT meant to enter (incomplete/never-onboarded org, no brand) bouncing to `/onboarding` is CORRECT (that's the first-run gate, not this bug) ŌĆö THIS bug is a blank page on a COMPLETE org (has brands). **Fix (KevinLourd/godmode-join-idempotent):** drop the paged pre-check; `createOrganizationMembership` directly and treat a Clerk error whose `errors[].code === "already_a_member_in_organization"` as success (`{ok:true}`) ŌĆö trust Clerk (server-side source of truth) over a paged list. Applied byte-equal to BOTH `apps/dashboard` + `apps/admin` join routes. Hardening in the same PR: `OrgActivator` now `await session?.getToken({skipCache:true})` AFTER `setActive` (mirrors `handleOrgSwitch`, CLAUDE.md DIS-143 item 3) to close the stale-token 409 window. **Diagnostic approach that nailed it:** the route only returned `err.message` ("Bad Request", Clerk's generic) ŌĆö the real `errors[].code` was in the **Vercel runtime logs** (`get_runtime_logs query:"admin-join"` ŌåÆ `already_a_member_in_organization`); and the "is this org actually complete?" question was answered by querying **brand-service** (`org_brands` for the org ŌåÆ 2 brands = onboarded ŌåÆ blank is a bug, not the onboarding gate). Org NAMES live only in Clerk (client-service `orgs.name` is null for everyone), and the prod `CLERK_SECRET_KEY` Backend API is IP-restricted from a dev machine (403 code 1010) ŌĆö so map a Clerk-org-name to data via brand-service, not Clerk. ## Release flow (distribute.you specifics)

No `release.sh hotfix` here (that targets Railway semver services). Vercel deploys on every `main` merge ŌĆö no tag, no bump.

**NEVER run `vercel --prod` or `vercel deploy --prod` locally.** This bypasses git entirely, deploys a local build directly to prod, and wipes the Vercel project settings (`rootDirectory`, `framework`, `buildCommand` ŌåÆ all `null`). This has caused 3+ prod outages (pricing 404, stuck alias, copy regressions). The guard hook blocks it. If you see a `2ŌĆō4s` "build" by `adam-atomic-git` in `vercel ls`, that's a local deploy ŌĆö run the recovery below. Always push to `origin` via PR and let Vercel auto-deploy from the merge commit.

**Conductor workflow ŌĆö create a new workspace per task, never `git checkout -b` manually.** Conductor creates each workspace as a git worktree on its own branch and fetches `origin` first, so the branch already starts from the latest remote commit. Branch prefix: `adam/<workspace-name>` (configured via `.conductor/settings.local.toml`). If the workspace has been open a while and `origin/main` has advanced, run `git fetch origin main` before creating a PR.

- **Hotfix** ŌåÆ branch from `origin/main`, PR ŌåÆ `main`, `gh pr merge --auto --squash`.
- **Bugfix / Feature** ŌåÆ branch from `origin/staging`, PR ŌåÆ `staging`. Promotion to `main` via the stagingŌåÆmain PR flow.
- **Exception (`apps/dashboard/` + `apps/landing/` only):** dashboard-only and landing-only PRs merge straight to `main` in recent practice. Touching ONLY those dirs ŌåÆ follow the last 5 merges on the file; when in doubt, staging. Cross-cutting (shared/, packages/, multi-app) MUST go through staging.
  - **A NEW app under `apps/` (anything other than dashboard/landing), or ANY edit to root `package.json` / `pnpm-lock.yaml` / `pnpm-workspace.yaml` / `turbo.json`, is cross-cutting ŌåÆ prefer STAGING by blast-radius POLICY (not a hook block).** ŌÜĀ’ĖÅ The `guard-branch.sh` protect-main hook does NOT inspect paths/lockfiles ŌĆö rule 5 only checks `git merge-base --is-ancestor origin/main "$CURRENT"` and ALLOWS `gh pr create --base main` for ANY branch descending from `origin/main`. So a new-app / root-manifest / lockfile PR to main is mechanically ALLOWED (verified #2113: removed `apps/app` + edited `pnpm-lock.yaml`, PR to main, hook passed). The staging routing is a deliberate blast-radius CHOICE for additive/cross-cutting work, NOT something the hook enforces ŌĆö and for an ISOLATED change (e.g. deleting one self-contained app) where a `promote` would drag unrelated divergent commits, main-direct is the CORRECT call. Don't refuse main-direct citing a hook block that doesn't exist; decide on real blast radius. (Stale prior claim from #1561 said the hook BLOCKS this ŌĆö it does not, at least on the current hook.)

### Local-first SWR cache (per-query IndexedDB persister) ŌĆö open a page ŌåÆ its content NOW

**This is the dashboard's named architecture: a LOCAL-FIRST, stale-while-revalidate (SWR) cache.** Local-first = the on-disk cache is the source the UI paints FIRST, network is secondary. SWR = show the cached (stale) data instantly, refetch in the background, swap fresh in silently (TkDodo: *"stale data is better than no data, because no data means a loading spinner = perceived slow"*). It is the 4th "don't revert resolved state on a transient" layer and the only one that survives cache EVICTION/RELOAD (the first three keep warm-in-memory state across refetch only): 1. `keepPreviousData` (data) ┬Ę 2. `useCoordinatedReveal` (reveal) ┬Ę 3. `useMonotonicStatuses` (row bucket).

**Mechanism (`src/lib/query-provider.tsx` + `src/lib/persist-cache.ts`, unit-tested):** the **PER-QUERY persister** `experimental_createQueryPersister` (from `@tanstack/react-query-persist-client`), wired as a default query option (`queries.persister = persister.persisterFn`) on a plain `QueryClientProvider`. Storage = **IndexedDB** via `idb-keyval`. NOT the old whole-client `PersistQueryClientProvider` + `createSyncStoragePersister` (localStorage).

**Why the migration (2026-06-25, KevinLourd/dashboard-instant-cache) ŌĆö the 2026-06-23 "persist-everything into localStorage" policy was SELF-DEFEATING, causing the exact two symptoms it meant to fix ("30s skeletons all the time" + "site lourd/lent"):**
- **localStorage's hard ~5 MB per-origin cap** ŌåÆ a heavy brand's `brandLeads`/`brandEmails` blew the cap ŌåÆ `removeOldestQuery` evicted the small overview queries (cached first) ŌåÆ the overview cold-skeletoned on the slow Neon chain. The persist-all policy evicted the very page it was meant to keep.
- **Whole-client `dehydrate()` re-serializes the ENTIRE cache on EVERY mutation** (TanStack #9775). ~12 overview queries polling 5s ŌåÆ constant multi-MB `JSON.stringify` + sync write on the MAIN THREAD ŌåÆ jank/"lourd".
- The per-query persister fixes BOTH: each query is written **separately, keyed by its query hash, only when IT changes** (no whole-cache re-serialize) AND IndexedDB has **no 5 MB cap + async off-main-thread writes**. Verified against TanStack `createPersister` docs + TkDodo #5169.

**`admin` ŌēĀ `dashboard`: separate Vercel apps = separate ORIGINS = separate storage.** `admin.distribute.you`'s heavy Outlets/Journalists/Articles cache NEVER touches the `dashboard.distribute.you` cache. Don't diagnose dashboard slowness via admin's big lists; the dashboard customer never even loads those (alpha/staff-gated). Admin is allowed to stay heavy.

**Load-bearing rules:**
- **`maxAge: Infinity` (disk = keep forever, never expire) + `gcTime: 30min` (memory only) are DECOUPLED.** This decoupling is the whole reason per-query beats whole-client: with the per-query persister a heap-GC'd query **stays on disk**, so "instant on open, any session, any day" needs NO `gcTime: Infinity` ŌåÆ no heap blowup. (The whole-client persister mirrors diskŌćåmemory, so `maxAge: Infinity` there would force `gcTime: Infinity` = the #1273 OOM ŌĆö impossible to have both there.) `PERSIST_MAX_AGE_MS = Infinity`, `PERSIST_GC_TIME_MS = 30min`. The old "`gcTime` MUST equal `maxAge`" rule (TanStack #5169) applies ONLY to the whole-client persister ŌĆö it does NOT bind here.
- **Org-scope via the persister `prefix`** = `persisterStorageKey(orgId)` (`distribute-dashboard-cache:{orgId}`); each query's storage key is `${prefix}-${queryHash}`, so org A's keys never collide with org B (DIS-143). **`storage: undefined` while `orgId` is null OR on the server** (`persistEnabled = typeof window !== "undefined" && !!orgId`) ŌĆö no anon-bucket cross-org bleed. The keyed remount (`key={orgKey}`) still resets the in-memory cache on switch.
- **`buster` = MANUAL version constant** (`persistCacheVersion()` ŌåÆ `PERSIST_CACHE_VERSION`, currently `"1"`), bumped BY HAND only on an incompatible shape change ŌĆö NEVER the git commit SHA (the SHA flips every deploy, ~12├Ś/day ŌåÆ wiped the disk cache on ~every visit, defeating persistence). With `maxAge: Infinity` there's no expiry safety net, so `buster` is the ONLY forced invalidation; cross-deploy shape drift is otherwise tolerated by `safeParse`/`z.coerce` on list readers + keep-last-good `structuralSharing`. Bump when you rename/remove a field on a response type consumed straight from cache with no safeParse guard (additive optional fields do NOT need a bump).
- **Persist filter = success + non-sensitive + ALLOWLISTED** via the persister `filters.predicate` ŌåÆ `shouldPersistQuery` (reused from the old `shouldDehydrateQuery`). `PERSISTABLE_QUERY_ROOTS` is the INVENTORY of every live non-sensitive root (big lists included); default-OFF for an unlisted root. Secrets EXCLUDED (`apiKeys`/`byokKeys`/`keySources` via `SENSITIVE_QUERY_ROOTS` ŌĆö key material never on disk).

**Irreducible:** the genuine FIRST-EVER load on an empty cache still waits on the cold Neon backend (~20-30s, features/api/runs scale-to-zero 0.25 CU). Killing that needs backend warm-up (other repos), NOT a dashboard cache change.

**Rule:** persistence is GLOBAL ŌĆö one per-query persister wired as a default query option, never a second provider / per-page persister. `experimental_createQueryPersister` is `experimental_`-prefixed (stable behavior, API may shift ŌĆö pin to the installed `@tanstack/react-query-persist-client` version). Keep `PERSISTABLE_QUERY_ROOTS` in lockstep with live queries. (#1273 small-allowlist origin; #1278/DIS-170 30min+allowlist; 2026-06-23 persist-all-localStorage regression; 2026-06-25 KevinLourd/dashboard-instant-cache = per-query IndexedDB + maxAge Infinity, the current design.)

### keep-last-good (`structuralSharing`, cache-write boundary) ŌĆö a SUCCESSFUL refetch returning VALID-but-degenerate data must not collapse the UI

5th "don't revert resolved state on a transient" layer ŌĆö and the ONLY one for a **successful** refetch that legitimately returns **worse** data. The other four all assume the new data is good: 1. `keepPreviousData` (key-change / errorŌåÆundefined) ┬Ę 2. `useCoordinatedReveal` (reveal decision) ┬Ę 3. `useMonotonicStatuses` (status downgrade *ordering* on a poll) ┬Ę 4. persisted cache (eviction). **The gap:** a 200 whose nullable field flips non-nullŌåÆnull (a cold Neon chain ŌĆö apiŌåÆfeaturesŌåÆworkflow/runs/email-gateway/brand, all scale-to-zero ŌĆö answers a poll/refocus refetch with zeroed unit costs or fewer rows while half-warming). `keepPreviousData` can NOT help ŌĆö the null IS the new valid query data, so React Query installs it and any UI **derived** off it (budget cards, a `disabled` gate, a badge, a $ amount) collapses mid-session.

**Industry-aligned fix (researched: TkDodo / TanStack docs / SWR all agree there is NO built-in for successful-but-worse ŌĆö you intercept at the cache-write boundary).** TanStack's hook is `structuralSharing(prev, next)` (SWR's is `compare`, Apollo's is a `merge` field policy). Helpers in `src/lib/keep-last-good.ts`: `keepLastGoodFields(prev, next, fields, label)` (per-field coalesce: keep prev's non-null when next nulls it) + `keepLastGoodList(prev, next, {keyFn, fields, label})` (keyed merge: per-item coalesce + RETAIN a prev item that vanished from next). Both `console.error` the suppressed downgrade (fail-loud ŌĆö a persistent real downgrade still logs; this is keep-resolved, NOT a silent fallback). Compose per query (e.g. `keepLastGoodWorkflowProjection` in `api.ts`, co-located with the type) and pass it as `structuralSharing` on the `useAuthQuery` ŌĆö protects EVERY consumer of that query at ONE place, not per-component (a per-component latch is opt-in-and-forgettable, which is exactly how the collapse shipped).

**Five load-bearing rules:**
- **OPT-IN per query, NEVER a global default.** A null can legitimately mean "deleted/gone"; blanket last-good would mask real deletions. Apply ONLY where a null means "transient/not-ready" (cold compute, in-flight aggregation), not "removed".
- **Put it at the QUERY (`structuralSharing`), not at each consumer.** One merge ŌåÆ cards + button + badges all inherit it. The per-component `useLastGoodValue` latch is strictly weaker (forgettable).
- **List ONLY the fields whose null = transient.** Fields outside the allowlist let next win (incl. when null) ŌĆö so a genuinely-cleared field still clears.
- **CUT the *steady-state* poll on econ-INDEPENDENT slow-changing queries ŌĆö but keep a BOUNDED cold-start warm-up poll.** The workflow-projection unit costs are "fetched ONCE" by design (#1528/#1536) ŌĆö a perpetual 5s `...pollOptions` on them only re-hits the cold Neon chain for zero steady-state benefit (cards update from the client `econLive` recompute). BUT "fetch-once + `refetchOnWindowFocus`" alone is NOT correct: **keep-last-good cannot heal the FIRST fetch** (no prior good value to keep), and the cold Neon chain most often returns its degenerate-200 (null unit costs / empty workflows) on exactly that first request ŌĆö so a cold load STICKS on the empty-state ("No workflow has website-visit cost data yet") until a manual window-refocus. The right shape is a **conditional `refetchInterval`**: poll (~4s) ONLY while the payload has no usable data (e.g. no workflow with non-null `costPerCloseUsd`), capped at N tries (so a genuinely-empty brand stops), `false` the instant it warms. This self-heals cold-start without perpetual polling. (#1557-regression: #1542 dropped the poll entirely calling it "pointless"; it was the cold-start self-heal ŌĆö keep-last-good #1548 only covers a later bad refetch of an already-good cache, never the first fetch.)
- **TanStack types `structuralSharing` params as `unknown`** ŌĆö pass `(prev, next) => keepLastGoodX(prev as T | undefined, next as T)`.

**Rule:** any **polled / refetch-prone** query with **nullable fields that GATE or compute UI** (disabled-button, empty-state, $ amount, recommended pick) MUST pass a keep-last-good `structuralSharing`. Pure read-display surfaces that already degrade to "ŌĆö" on null are lower-risk (phased-sweep, not a hotfix). (#1542-regression: the launch-page budget cards + Launch button ŌĆö then `campaigns/new`, now `brands/[brandId]/launch` ŌĆö collapsed when the polled `workflowProjection` caught a valid-but-degenerate 200 mid-session; `econLive`/┬¦2 inputs were NOT the cause ŌĆö they're latched. The poll predated #1542; #1542 widened it by recomputing cpc client-side. Fix: keep-last-good `structuralSharing` on the query + drop the pointless 5s poll. ### Org switching ŌĆö cross-org isolation framework (defense-in-depth)

4th "don't act under the wrong context" member (after data/reveal/bucket). Keeps every API call + the whole cache bound to the viewed org. **Failure:** the proxy forwards the Clerk session active org (`auth().orgId`), decoupled from the URL `/orgs/[orgId]` (Clerk `organizationSyncOptions` syncs URLŌåÆsession on page routes, NOT `/api/*`); after `setActive` the JWT rotates async ŌåÆ a lag window carrying the OLD org ŌåÆ a stale write commits under the old org (later 404) or a stale poll 404s. campaign-service isolation is correct; the 404 is the symptom. (DIS-143). Doctrine converges (Clerk + OWASP Multi-Tenant + tkdodo): derive tenant from JWT claims not client headers, validate every request, fail CLOSED, remount on switch.

Framework (every layer central, one file ŌĆö never per-page):
1. **Server choke point ŌĆö fail-closed proxy guard** (`src/lib/proxy-org.ts` `checkProxyOrg`): the client sends `x-active-org-id` (the URL org); the proxy compares it to `auth().orgId`, returns **409 `org_desync`** on mismatch, refuses to forward. The JWT stays the sole authority (client value detects the desync only ŌĆö OWASP fail-closed, not trust-the-header). Applied to ALL proxy routes; a new route ŌåÆ call `checkProxyOrg` right after the `clerkOrgId` check. Constants in dependency-free `src/lib/org-desync.ts`.
2. **Client cache choke point ŌĆö keyed remount** (`src/lib/query-provider.tsx`): inner `QueryClientProvider key={org.id}` ŌåÆ org switch = new client = empty cache atomically. REPLACES `queryClient.clear()` (raced). So `OrgCacheInvalidator` MUST mount ABOVE `QueryProvider` (survives the remount); it no longer touches the cache NOR navigates ŌĆö it ONLY clears breadcrumb caches on an org change (the cross-tab `router.push` was removed in #1944, see item 5). **The remount key MUST be MONOTONIC ŌĆö latch it to the last RESOLVED org id, never raw `orgId ?? "no-org"`.** Clerk's `useOrganization()` blinks `organization: null` transiently during background JWT rotation (~1/min) + tab focus/reconnect; a raw key flips `realIdŌåÆ"no-org"ŌåÆrealId` on every blink ŌåÆ spurious remount ŌåÆ empty in-memory cache. Persisted queries rehydrate from IndexedDB silently (ALL live roots, big lists included, no 5MB cap since the 2026-06-25 per-query IndexedDB migration ŌĆö so this blink-skeleton is gone). Fix: `const lastOrgId = useRef(null); if (orgId) lastOrgId.current = orgId;` then key + `orgId` prop both off `lastOrgId.current` ŌĆö a null blink keeps the last key (no remount); a real switch to a DIFFERENT org still advances the id ŌåÆ remount + fresh per-org persister prefix, DIS-143 intact. Same monotonic-gate rule as the app-shell auth latch. (#1767)
3. **Source ŌĆö `await setActive` THEN `await session?.getToken({ skipCache: true })`** (`breadcrumb-nav.tsx` `handleOrgSwitch`) before `router.push`. `await setActive` alone is NOT enough: it resolves on the CLIENT before its `Set-Cookie` propagates, so the very next navigation reaches `proxy.ts` `organizationSyncOptions` carrying the STALE token (active = previous org / not-a-member of the target). Clerk bounces the URL back ŌåÆ `OrgActivator` re-syncs the client to the previous org ŌåÆ **the switch reverts on its own** (loudest in STAFF god-mode, where the target is a just-joined customer org whose membership the stale JWT lacks). The `getToken({ skipCache: true })` re-mint forces the fresh token (new active org + new membership) into the cookie BEFORE the middleware reads it ŌĆö same proven fix as `beta-onboarding`'s onboarding-complete hop (CLAUDE.md "Stale token ŌĆö force `getToken({ skipCache: true })` before navigating"). Guarded by `org-switch-cache.test.ts` (getToken must sit between setActive and router.push). (#1940)
4. **Read gate** (`src/lib/use-auth-query.ts`): `enabled: caller && urlOrg === activeOrg` (strict `!urlOrgId || urlOrgId === activeOrgId` ŌĆö NO `|| !activeOrgId` escape; a read under a null active org lands in a null-org cache entry, TanStack #3743). `useOrganization` import allowed here (powers the gate, not token-passing).
5. **MULTI-TAB ŌĆö scope every request by the PER-TAB token, NEVER the shared cookie; never navigate a tab on the shared active-org signal.** Clerk supports a per-tab active org, BUT **the session cookie is one global singleton for the whole browser** ŌĆö it reflects whichever tab was focused last ([Clerk: Organizations ŌåÆ "multiple browser tabs"](https://clerk.com/docs/guides/organizations/overview)). So scoping data off the cookie (`auth()` with no Bearer) makes a background poll / navigation from a NON-focused tab run under the WRONG org ŌåÆ cross-org bleed + 409 churn; and any code that NAVIGATES a tab when the (shared) `useOrganization()` active-org changes yanks tab A onto tab B's org the instant tab B switches ŌĆö the "org switches by itself across tabs" bug. Two-part fix (#1944), per Clerk's own multi-tab guidance + WorkOS/SuperTokens "the token IS the tenant context":
   - **Data path:** `api.ts` `apiCall` attaches THIS tab's token ŌĆö `Authorization: Bearer ${await window.Clerk.session.getToken()}` (via `getTabSessionToken()`, a non-hook read of the per-tab `window.Clerk` client) ŌĆö on every `/api/v1/*` proxy call. Clerk's `auth()` honors a Bearer over the cookie ([Clerk: making authenticated requests](https://clerk.com/docs/guides/development/making-requests)), so the proxy scopes off the per-tab org. Optional-chained ŌåÆ before Clerk loads / signed-out it falls back to the cookie, and `checkProxyOrg` (item 1) still fails closed, so there's NO new exposure even if a Bearer is ever absent.
   - **Navigation path:** `OrgCacheInvalidator` NEVER navigates (removed `router.push`/`useRouter`/`usePathname`). The URL `/orgs/[id]` is the per-tab source of truth; in-tab switches navigate via the breadcrumb (`handleOrgSwitch`), URLŌåÆactive alignment via `OrgActivator`. Do NOT re-add a navigate-on-active-org-change effect ŌĆö it re-introduces the cross-tab self-switch.
   - **Display / cache-key path (#1948 ŌĆö the one that actually stops the VISIBLE oscillation):** #1944's data+nav fixes were necessary but NOT sufficient ŌĆö the org name kept flipping between tabs because the CLIENT UI still keyed on `useOrganization()` (the shared active org that flips cross-tab). Every per-tab client surface must key on the **URL org** (`/orgs/[id]`), never the active org: (a) `query-provider.tsx` remount key = `urlOrgId ?? organization?.id` (URL is per-tab-stable ŌåÆ no remount storm when a sibling tab switches; falls back to active org only OFF the /orgs/ tree, e.g. onboarding); (b) `breadcrumb-nav.tsx` renders the URL org via a per-orgId display cache (`orgDisplayCacheRef`, populated when `organization.id === orgId`) so the name/avatar never flip; dropdown highlight compares `orgId === o.id`, not the active org. RULE: in a multi-tab org app, treat the URL as the per-tab source of truth for ALL of {data scope, cache key, displayed org, nav}; `useOrganization()` is shared/global and may only feed (i) the cross-tab-stable parts via a per-URL-org cache, or (ii) off-/orgs/ fallbacks. Don't bind a per-tab surface directly to `useOrganization()`.
   - Guarded by `org-switch-cache.test.ts` (per-tab Bearer guard + no-cross-tab-navigation guard + URL-keyed remount guard) + `breadcrumb-hierarchy`/`feature-overview-polish`/`last-brand` (URL-org display). Note the cookie-era item 3 (`getToken({skipCache})` after `setActive`) is STILL correct for the focused-tab navigation; item 5 makes the DATA + DISPLAY paths immune to the cookie regardless of focus.

**Self-heal:** on 409 `org_desync`, `apiCall` waits 500ms + retries once. **Rule:** JWT authority server-side every request fail-closed; cache resets by remount not `clear()`; remount-surviving nav lives above `QueryProvider`. (DIS-143)

### Public marketing pages ŌĆö SEO + AI-scraper rendering

For any public landing/marketing/docs page (anything under `apps/landing/`, `apps/sales-cold-emails-landing/`, `apps/docs/`), the default rendering strategy is **ISR + `unstable_cache` + edge cache**, NEVER Suspense streaming on indexable content.

Why: Next.js streaming SSR injects content into `<template>` blocks resolved by client-side JS. Googlebot (Chromium-headless) executes JS and sees the resolved content, but **AI scrapers (GPTBot, ClaudeBot, PerplexityBot, Bingbot AI, curl-like crawlers) parse raw HTML only** ŌĆö they receive the skeleton fallback, not the data. Putting leaderboards, benchmarks, blog lists, stats behind `<Suspense>` kills AI Search indexing for the very content that earns AI Search traffic.

Concrete rules:

1. **No `<Suspense>` on indexable content.** Hero copy, h1/h2, body text, JSON-LD blocks, leaderboards, benchmark tables, blog cards, stats ŌĆö all SSR-sync in the byte stream. Suspense reserved for user-specific, non-indexable widgets (logged-in dashboards, real-time counters, chat).
2. **Never combine `force-dynamic` with `revalidate`.** `force-dynamic` wins and ISR is silently disabled. Prefer ISR (`export const revalidate = N`) alone; let `unstable_cache` + graceful build-time empty-result handle the missing-config case.
3. **Calling `headers()` or `cookies()` in a page forces dynamic rendering** even with `revalidate` set. Drop them if the page can be ISR-prerendered (resolve hostname-dependent URLs from `NEXT_PUBLIC_*` env vars instead).
4. **Wrap every fetch / DB call used by a public page in `unstable_cache`** with explicit tags. Edge cache (`s-maxage / stale-while-revalidate`) provides instant HTML; `unstable_cache` provides instant data on the rebuild after invalidation. Both stack.
5. **Invalidate via `revalidateTag(tag, profile)`** in the publish/update flow (webhooks, admin forms). Next 16.2+ requires the 2nd arg `profile: 'default' | 'seconds' | 'minutes' | 'hours' | 'days' | 'weeks' | 'max' | CacheLifeConfig`. Pass `"default"` for ordinary invalidations.
6. **`unstable_cache` at module top level breaks vitest** because the wrapping runs at import time before `vi.doMock` can intercept. Mock `next/cache` globally via `apps/landing/tests/setup.ts` (`unstable_cache` ŌåÆ pass-through identity, `revalidatePath` / `revalidateTag` ŌåÆ `vi.fn()`); declare `setupFiles: ["tests/setup.ts"]` in `vitest.config.ts`.
7. **Validate scraper output after any caching change** with `apps/landing/scripts/seo-snapshot.sh [BASE_URL]` (counts `<h1>`, `<h2>`, `<article>`, JSON-LD blocks under a `GPTBot` user-agent) and `apps/landing/scripts/measure-ttfb.sh [BASE_URL]` (warm/cold TTFB per route). Both pre- and post-change snapshots in the PR description.

Incident 2026-05-24 (distribute.you#1153): pre-refactor, `/blog` TTFB was 1.5ŌĆō2.3s because `force-dynamic` overrode `revalidate = 60` and every visit hit Neon directly. `/benchmarks` was dynamic because `headers()` was called for hostname resolution. `/investors` took 6ŌĆō8s end-to-end because metrics were `cache: "no-store"`. Three loading.tsx files existed but were never reachable (page components weren't async-suspended). Single PR flipped 8+ routes from `ŲÆ Dynamic` to `ŌŚŗ Static` ISR with `unstable_cache` + tag invalidation; no Suspense or skeleton was added on any indexable content.
